# Hakamap 開発作業規約

## 文書の正本

- 製品概要、開発方法、および文書への入口は`README.md`を参照します。
- 利用者視点の機能要件は`docs/requirements`を正とします。
- 技術方式、保存方式、性能、復旧、およびログは`docs/design`を正とします。
- 重要なアーキテクチャ判断は`docs/adr`へ記録します。
- 作業再開時は`docs/development-handoff.md`と現在のGit差分を確認します。
- 文書間で矛盾する場合は正本とGit差分を確認し、判断できなければ推測で確定せず
  ユーザーへ確認します。

## 変更してはならない基本制約

- Hakamapはオフラインかつ単一ユーザー向けのWindowsアプリです。
- 実行時にクラウド同期、テレメトリー、外部地図タイル、CDN、その他のネットワーク依存を
  追加しません。
- HTTPサーバーは`127.0.0.1`だけにバインドし、ReactとAPIを同一オリジンで提供します。
- UIはReact、TypeScript、PixiJS、バックエンドはJava 21とSpring Bootを使用します。
- Electron、Tauri、およびJavaFXを追加しません。
- 永続化はプロジェクトフォルダ内のJSONとアセットファイルを使用します。H2、JPA、Flywayを
  新しい設計へ持ち込まず、既存雛形に残る依存はJSON実装への移行時に除去します。
- 保存データの正はJavaバックエンドとし、フロントエンドへ業務ロジックを重複実装しません。
- ユーザーデータ、生成DB、背景、添付、バックアップ、復旧データをGitへコミットしません。
- 個人情報、ユーザー入力、ファイル内容、絶対パスを通常ログへ出力しません。

## 実装前の確認

- 既存コード、プロジェクト構成、対象ディレクトリ内の指示を確認します。
- 要件と基本設計が確定しているか確認し、未決事項を実装判断だけで確定しません。
- 既存の設計、命名、ディレクトリ構成を優先します。
- 要件と無関係な大規模リファクタリングを行いません。
- 新しい依存関係は必要最小限とし、Java 21との互換性、オフライン動作、Windowsへの同梱、
  デスクトップ配布可能なライセンスを確認します。
- 製品上の重要な決定または変更は、該当する要件・設計文書とADRへ反映します。

## フロントエンド規約

- ReactおよびTypeScriptはAirbnb JavaScript Style Guideに従います。
- TypeScriptのstrictモードを使用し、`any`を避けます。
- 標準UIにはMUIとEmotionを使用し、共通値はテーマで管理します。
- UIローカル状態にはZustand、サーバー状態にはTanStack Queryを使用します。
- クライアント境界の検証にはZodを使用します。
- PixiJSを専用アダプター内へ閉じ込め、Reactコンポーネント全体へ拡散させません。
- ReactはUI、PixiJSは地図キャンバスを担当します。
- キーボード、ポインター、視認可能なフォーカス、およびアクセシブルなフォームに対応します。
- CDNからフォント、アイコン、スクリプト、地図タイル、その他の実行時アセットを取得しません。

## バックエンド規約

- JavaコードはGoogle Java Style Guideに従います。
- 機能またはドメイン単位で構成し、コントローラーを薄く保ちます。
- API境界ではDTOを使用し、保存用JSONモデルを直接公開しません。
- 可能な場合、変更不能なDTOにはrecordを使用します。
- コンストラクターインジェクションを使用し、フィールドインジェクションを禁止します。
- 状態を変更するリクエストをサーバー側で検証し、Problem Details形式でエラーを返します。
- ファイル操作には`java.nio.file.Path`と構造化APIを使用します。
- パストラバーサル、リンク、および管理領域外へのアクセスを拒否します。
- APIレスポンスへ絶対パスや内部実装を含めません。

## テストと検査

- Javaコード変更後は、Checkstyle、Spotless、および関連テストを実行します。
- Reactコード変更後は、ESLint、Prettier、および関連テストを実行します。
- バックエンドの標準検査は`cd backend && ./mvnw verify`です。
- フロントエンドの標準検査は`cd frontend`で`pnpm lint`、`pnpm format:check`、`pnpm test`、
  `pnpm build`を実行します。
- ファイルシステムを扱うテストでは一時ディレクトリを使用し、実際のユーザーデータ領域を
  読み書きしません。
- 不具合修正時は合理的に可能な限りリグレッションテストを追加します。
- 検査エラーを無視して完了扱いにしません。実行できない検査はコマンドと理由を報告します。

## 変更時の規律

- オフライン動作とローカルデータの安全性を守れる、最も単純な実装を優先します。
- 生成物をバージョン管理へ含めません。
- 設定変更の理由を作業結果で説明します。
- ユーザーの既存変更を保持し、無関係な差分を巻き戻しません。
- ユーザーの指示なしにコミットまたはプッシュしません。
- Hakamap MVP実装では、ユーザーが承認した`docs/implementation-plan.md`のPhase単位で
  実装、検査、関連文書、および引継ぎメモをまとめてコミットします。Phase途中の作業は
  原則としてコミットせず、再開に必要な状態は引継ぎメモと作業ツリーで管理します。
- Phaseコミットは、当該Phaseの完了条件と標準検査を満たした後に作成します。未完了や
  検査失敗が残る場合は完了コミットとして扱わず、理由を報告します。
- Phaseコミットを作成しても自動的にプッシュしません。外部のChatGPTレビューを行う節目、
  またはユーザーが明示的に指示した場合だけプッシュします。
- ChatGPTレビュー用にプッシュした場合は、`git rev-parse HEAD`と
  `git rev-parse origin/<現在のブランチ>`の一致を確認し、レビュー対象となる
  Phaseコミットの完全なコミットIDまたは一意な短縮IDをユーザーへ明示します。

## 作業の引き継ぎ

- ユーザーがDev Containerまたはコンテナをリビルドすると伝えた場合は、リビルド前に
  `docs/development-handoff.md`を更新します。
- ユーザーが一日の作業を終了すると伝えた場合も、終了前に引継ぎメモを更新します。
- 引継ぎメモには現在の目的、完了事項、未完了事項と次の手順、変更ファイル、検査結果、
  既知の問題や注意事項を記載します。
- 引継ぎメモには、レビューまたは作業再開の基準となる、作業成果を含む直近の
  プッシュ済みコミットIDを記載します。コミットまたはプッシュ後は`git rev-parse HEAD`と
  `git rev-parse origin/<現在のブランチ>`を照合し、基準コミットIDと反映先を更新します。
- 引継ぎメモ自身を変更すると新しいコミットIDが生じるため、このIDを「リポジトリの
  最新コミット」と表現せず、「レビュー基準コミット」または「作業再開基準コミット」と
  表現します。引継ぎメモだけを修正した後続コミットは、作業成果の基準IDを変更する理由に
  しません。
- 引継ぎメモへ秘密情報、個人情報、ユーザーデータ、不要な会話全文を記録しません。
- リビルド後は引継ぎメモとGit差分を最初に確認し、実際のワークツリーを正として継続します。
- 作業終了時に変更がコミット・プッシュされていない場合は、外部のChatGPTから参照できるよう
  コミットとプッシュをユーザーへ提案します。
- 外部のChatGPTとの検討内容を受け取った場合は、引継ぎメモ、現在の文書、Git差分と照合し、
  確定内容を該当文書またはソースへ反映します。
- 外部連携内容と現在の要件・実装が矛盾し、採用判断ができない場合は上書きせず確認します。

## 使用言語

- ユーザーとの会話、進捗、質問、作業結果は日本語で記述します。
- コミットメッセージとGitHub上のコミットコメントは日本語で記述します。
- ソースコード内のコメントとJavadocは日本語で記述します。
- README、要件、設計、ADR、開発手順、およびUI文言は日本語で記述します。
- 識別子、APIフィールド、設定キー、コマンド、および外部仕様の文字列は一般的な慣習に従い
  必要に応じて英語を使用します。
- コミットメッセージは、変更内容と目的が日本語で簡潔に分かる表現にします。

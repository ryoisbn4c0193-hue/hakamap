# Hakamap 開発ガイド

## 製品概要

Hakamapは、独自の2D墓地マップを作成し、地図上の場所に墓所関連の
情報を紐づける、オフラインかつ単一ユーザー向けのWindowsアプリです。

このアプリは、次の要件を満たします。

- Windows 11上でローカル実行できること。最終的なインストーラー形式は基本設計で決定します。
- インターネット接続や外部サービスなしで動作すること。
- MVPでは背景、長方形のエリア、および長方形の墓所を作成・編集できること。
- 通路、入口、建物、自由地図要素、および地図要素種類マスターはMVP対象外とし、
  将来拡張候補として扱うこと。
- すべてのユーザーデータをローカルに永続化し、バックアップと復元ができること。
- 画面の実行方式と起動方法は、最終採用技術とともに基本設計で決定すること。
- 氏名、埋葬記録、メモ、写真、添付ファイルを非公開のローカルデータとして扱うこと。

ユーザーが製品要件を明示的に変更しない限り、クラウド同期、テレメトリー、
外部の地図タイル、CDNなど、実行時に必要となるネットワーク依存を追加しません。

## 現在のリポジトリ構成と技術候補

次のトップレベルモジュールを持つモノレポとして構成します。

```text
frontend/    ReactおよびTypeScriptクライアント
backend/     Spring BootアプリケーションとDBマイグレーション
packaging/   Windows向けjpackage設定とパッケージ作成スクリプト
docs/        アーキテクチャ決定記録と製品ドキュメント
```

現在の開発用スキャフォールドでは次の技術を使用していますが、製品の最終採用技術、
アプリケーションのレイヤー構成、および配布方式は基本設計で再確認します。変更が
決定するまでは、既存コードの変更に次の規約を適用します。

- フロントエンド: React、TypeScript、Vite。パッケージ管理にはpnpmを使用。
- 地図の描画と編集: PixiJS。
- サーバー: Java 21 LTS、Spring Boot。
- ビルドシステム: Maven Wrapper。グローバルにインストールしたMavenへ依存しないこと。
- 永続化: Spring Data JPA、組み込みファイルモードのH2、Flyway。
- フロントエンド状態管理: UIのローカル状態にはZustand、サーバー状態にはTanStack Query。
- バリデーション: サーバーではBean Validation、クライアント境界ではZod。
- パッケージ作成: Windows上で`jlink`と`jpackage`を使用。

Tauri、Electron、JavaFXを含むデスクトップ実行方式の最終選定は未確定です。現在の
スキャフォールドはSpring Bootがフロントエンドを配信し、既定ブラウザで開く方式です。

## 現行スキャフォールドの実行時境界

- HTTPサーバーは`127.0.0.1`だけにバインドし、全ネットワークインターフェースには公開しません。
- Reactの本番ビルドをSpring Bootから配信し、UIとAPIを同一オリジンにします。
  制限のないCORS設定は禁止します。
- 空いているポートを自動選択し、アプリの準備完了後にのみブラウザを開きます。
- 開発モードは分離し、ViteからSpring Bootの`/api`へプロキシして構いません。
- PCが完全にオフラインでも、すべての機能を利用可能にします。
- 2つ目のネイティブバックエンドを追加したり、フロントエンドに業務ロジックを
  重複実装したりしません。

## データ保存

- プロジェクトの保存場所は利用者がプロジェクトごとに指定できるようにします。
- Windowsからアクセス可能なローカルディスク、外付けストレージ、および
  ネットワークドライブを対象とします。
- プロジェクトを単一ファイルにするかフォルダ構成にするか、構造化データにH2、JSON、
  その他の形式のどれを使用するか、および添付ファイルを外部保持または埋め込みに
  するかは基本設計で決定します。
- `%LOCALAPPDATA%/Hakamap`は、実行時情報、自動復旧データ、一時ファイル、および
  ログなどのアプリケーション管理領域として使用します。
- バックアップ、復元、インポート、エクスポートでは、構造化データと参照先アセットを
  1つの整合性単位として扱います。
- 生成されたデータ、ユーザーの地図、添付ファイル、バックアップをGitへコミットしません。

## ドメイン設計

ドメインの概念を、コントローラーやReactコンポーネントから独立させます。
初期のドメイン用語は次のとおりです。

- 墓地プロジェクト: 墓地マップとその設定をまとめた自己完結した単位。
- 墓地マップ: 寸法、座標系、背景、編集可能な敷地を持つ地図。
- 区画: 墓地内にある名称または番号の付いた範囲。
- レイヤー: 敷地、通路、墓域、ラベル、目印など、表示順、表示状態、
  ロック状態を持つ将来拡張候補の地図要素。
- 墓所マーカー: 地図上の位置、大きさ、回転角度、区画参照、表示ラベル、外観、
  墓所記録との関連を持つ長方形の要素。
- 人物情報: 墓所マーカーに関連づけるローカル記録。1つの墓所に複数人を関連付けられる
  ものとし、具体的な入力項目と必須条件は今後の要件定義で決定します。
- 場所ピン: 入口、施設、樹木、記念碑、基準点など、墓所以外の将来拡張候補の地点。
- アセット: プロジェクトから参照されるローカル保存された画像や添付ファイル。

座標は画面ピクセルではなく、地図座標として保存します。描画変換、ズーム、パンは
フロントエンドの責務とし、保存座標を変更してはいけません。プロジェクトを
エクスポートできるよう、保存データは端末固有の絶対パスへ不要に依存させません。

墓の形状、家族構成、氏名形式、日付形式、埋葬方法、区画番号について、特定の文化に
依存する前提を置きません。要件が確定していない項目は任意入力とし、分類を拡張可能にします。

## API規約

- HTTPエンドポイントは`/api`配下に配置します。
- API境界ではDTOを使用し、JPAエンティティを直接シリアライズしません。
- 状態を変更するすべてのリクエストをサーバー側で検証します。
- エラーは一貫したProblem Details形式で返します。
- ファイルシステムのパスや実装詳細をAPIレスポンスへ含めません。
- 汎用CRUDよりも、ドメインの不変条件を守るまとまりのある操作を優先します。

## フロントエンド規約

- ReactおよびJavaScript／TypeScriptのコーディングは、Airbnb JavaScript Style Guideに従います。
- TypeScriptのstrictモードを使用し、`any`を避けます。
- アプリケーションUIの標準コンポーネントにはMUIのMaterial UIを使用し、
  独自コンポーネントを作る前に既存のMUIコンポーネントで実現できるか検討します。
- Material UIのスタイルエンジンには標準のEmotionを使用し、テーマで色、タイポグラフィ、
  余白、フォーカス表示などの共通デザイントークンを管理します。
- 地図描画コードを専用アダプター内に閉じ込め、PixiJSオブジェクトをReact
  コンポーネント全体へ拡散させません。
- ReactはアプリケーションUIを、PixiJSはインタラクティブな地図キャンバスを担当します。
- キーボードとポインター操作、視認可能なフォーカス、アクセシブルな墓所情報フォームに対応します。
- 地図編集のUndoとRedoは、明示的なドメインコマンドとして扱います。
- CDNからフォント、アイコン、スクリプト、地図タイル、その他のアセットを取得しません。
- すべての実行時アセットをアプリまたはユーザープロジェクトに同梱します。

## バックエンド規約

- Javaのコーディングは、Google Java Style Guideに従います。
- バックエンドコードは、コントローラー、サービス、リポジトリをそれぞれ1か所へ
  集約するのではなく、機能またはドメイン単位で整理します。
- コントローラーを薄く保ち、トランザクションはアプリケーションサービスで管理します。
- 可能な場合、変更不能なリクエストDTOとレスポンスDTOにはrecordを使用します。
- コンストラクターインジェクションを使用し、フィールドインジェクションは禁止します。
- ファイル操作には`java.nio.file.Path`と構造化されたAPIを使用します。
- プロジェクトのアセットパスを、パストラバーサルやデータディレクトリ外への
  アクセスから保護します。
- 運用上必要なイベントはログへ記録しますが、ユーザーが入力した非公開情報は記録しません。

## テスト

次のテスト層を用意します。

- ドメインルールに対するバックエンドのユニットテスト。
- 永続化、Flywayマイグレーション、APIの振る舞い、ファイルシステム境界に対する
  Spring統合テスト。
- VitestとReact Testing Libraryによるフロントエンドのユニットテストとコンポーネントテスト。
- 統合アプリの実行後に、重要な操作フローを確認するPlaywrightテスト。

テストでは一時ディレクトリと、採用する永続化方式に応じた隔離済みテストデータを使用します。
実際のユーザーデータディレクトリを読み書きしてはいけません。不具合修正時は、合理的に可能な限り
リグレッションテストを追加します。

## ビルドとパッケージ作成

- Mavenの本番ビルドでフロントエンドもビルドし、その出力をSpring Bootの
  静的リソースへ自動配置します。
- 開発時はフロントエンドとバックエンドを個別に起動して構いません。
- JavaScript依存関係は、リポジトリのロックファイルで固定します。
- Maven WrapperとNodeパッケージマネージャーのバージョンをリポジトリで管理します。
- 最終的なインストーラー形式、同梱ランタイム、およびパッケージ作成方式は基本設計で決定します。
- 現在の`jlink`および`jpackage`構成は技術候補として扱います。
- 自動アップデートの有無は未確定とします。

## 開発環境

Dev ContainerにはJava 21、Node.js LTS、pnpm、
Git、一般的なシェルツールを用意します。ViteとSpring Bootの開発用ポートは公開しますが、
本番環境では固定ポートに依存しません。

推奨するVS Code拡張機能は次のとおりです。

```text
vscjava.vscode-java-pack
vmware.vscode-spring-boot
vscjava.vscode-spring-initializr
vscjava.vscode-maven
dbaeumer.vscode-eslint
esbenp.prettier-vscode
vitest.explorer
ms-playwright.playwright
yoavbls.pretty-ts-errors
redhat.vscode-xml
```

現在のH2スキャフォールド用にデータベースツールを追加する場合は、JDBC経由でH2に
対応しているものを選びます。SQLite専用ビューアーではH2ファイルを参照できません。

## 変更時の規律

- 実装前に既存コード、プロジェクト構成、対象ディレクトリ内の指示を確認します。
- 既存の設計、命名、ディレクトリ構成を優先します。
- 要件と無関係な大規模リファクタリングを行いません。
- 新しい依存関係は、目的達成に必要な最小限にします。
- オフライン動作とローカルデータの安全性を守れる、最も単純な実装を優先します。
- 重要なアーキテクチャ変更は`docs/adr/`へ記録します。
- 生成物をバージョン管理へ含めません。
- 製品上の決定によって制約が変わった場合は、このファイルを更新します。
- 新しい依存関係を追加する前に、実行時のオフライン動作と、デスクトップアプリとして
  配布可能なライセンスであることを確認します。
- Javaコードの変更後は、Checkstyle、Spotless、関連するテストを実行します。
- Reactコードの変更後は、ESLint、Prettier、関連するテストを実行します。
- 検査やテストのエラーを無視して、作業を完了扱いにしません。
- 設定を変更した場合は、その理由を作業結果で説明します。
- 実行できなかった検査がある場合は、対象コマンドと理由を明記します。

## 作業の引き継ぎ

- ユーザーがDev Containerまたはコンテナをリビルドすると伝えた場合は、リビルド前に
  `docs/development-handoff.md`へ作業状況を記録します。
- ユーザーが一日の作業を終了すると伝えた場合も、その日の作業終了前に
  `docs/development-handoff.md`を最新の状態へ更新します。
- 引き継ぎメモには、現在の目的、完了した作業、未完了の作業と次の手順、変更したファイル、
  実行した検査と結果、既知の問題や注意事項を記載します。
- 引き継ぎメモには、秘密情報、個人情報、ユーザーデータ、不要な会話全文を記録しません。
- リビルド後に作業を再開する場合は、最初に引き継ぎメモと現在のGit差分を確認し、
  実際のワークツリーを正として作業を継続します。
- ユーザーが作業を終了しようとしており、その日の変更がコミットおよびリモートへ
  プッシュされていない場合は、外部のChatGPTから最新情報を参照できるよう、
  コミットとプッシュを行うことを提案します。ユーザーの指示なしにコミットや
  プッシュを実行しません。
- ユーザーが外部のChatGPTと検討した内容をテキストで連携した場合は、最初に
  引き継ぎメモ、現在の要件文書、およびGit差分と照合し、確定した内容を該当する
  ソースまたは文書へ反映します。
- 外部のChatGPTから連携された内容と現在の要件または実装が矛盾し、どちらを採用するか
  判断できない場合は、既存内容を上書きせずユーザーへ確認します。

## 使用言語

- ユーザーとの会話、進捗報告、質問、作業結果の説明は日本語で行います。
- Gitのコミットメッセージと、GitHub上のコミットコメントは日本語で記述します。
- ソースコード内のコメントとJavadocは日本語で記述します。
- README、設計書、ADR、開発手順など、プロジェクト独自の文書は日本語で記述します。
- UIに表示する文言は、別途多言語対応の要件が決まるまで日本語を使用します。
- クラス名、メソッド名、変数名、ファイル名などの識別子は、一般的な開発慣習に従って
  英語で記述します。
- APIのフィールド名、設定キー、コマンド、ライブラリ名、拡張機能ID、外部仕様で
  定められた文字列は、必要に応じて英語のまま使用します。
- 英語の定型プレフィックスを必須とする規約は採用しません。コミットメッセージは、
  変更内容と目的が日本語で簡潔に分かる表現にします。

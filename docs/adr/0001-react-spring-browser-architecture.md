# ADR 0001: React・Spring Boot・既定ブラウザ構成を採用する

## 状態

採用

## コンテキスト

Hakamapは、Windows 11でローカル実行する単一ユーザー向けアプリケーションである。
完全オフライン動作、5,000墓所規模の2D地図編集、任意保存先へのプロジェクト保存、
バックアップ、および自動復旧を必要とする。既存リポジトリにはReact、TypeScript、
PixiJS、Java 21、およびSpring Bootのスキャフォールドが存在する。

## 決定

- UIにReactとTypeScriptを使用する。
- 地図描画にPixiJSを使用する。
- 業務ロジックとファイル操作にJava 21を使用する。
- ローカルAPIと本番UI配信にSpring Bootを使用する。
- UIは利用者の既定ブラウザで表示する。
- ローカルHTTPサーバーは`127.0.0.1`だけに公開し、UIとAPIを同一オリジンにする。
- Tauri、Electron、およびJavaFXは採用しない。
- Windows配布には`jpackage`を第一候補とする。

## 理由

- 既存スキャフォールドを活用できる。
- PixiJSによる地図編集とReactによる業務画面を分離できる。
- Java側で業務ルール、任意保存先へのファイル操作、バックアップ、および復旧を一元管理できる。
- Electronのように別のブラウザ実行環境を同梱せず、利用者の既定ブラウザを利用できる。
- UIとAPIを同一オリジンにすることで、実行時のネットワーク境界を単純化できる。

## 影響

- ブラウザから任意のローカルパスを直接操作できないため、Java側にネイティブファイル
  選択ダイアログとの連携が必要になる。
- ローカルHTTP APIを、同一端末上の外部サイトや別プロセスから保護する必要がある。
- バックエンドの起動完了を待ってからブラウザを開くランチャーが必要になる。
- ブラウザを閉じてもバックエンドを安全に終了する別の操作が必要になる。

## 採用しなかった選択肢

- Electron: ChromiumとNode.jsの同梱により配布サイズと更新対象が増える。
- Tauri: Rustによる2つ目のバックエンドと新しい技術境界が加わる。
- JavaFX: PixiJSと既存React画面を再利用できず、UIを再実装する必要がある。

# ADR 0002: プロジェクトフォルダへJSON形式で保存する

## 状態

採用

## コンテキスト

Hakamapは単一ユーザー向けで、1プロジェクトあたり5,000墓所を想定する。利用者による
手動保存、保存せず破棄、Undo・Redo、自動復旧、任意保存先への配置、およびプロジェクト
単位の移動とバックアップを必要とする。

既存スキャフォールドにはH2、Spring Data JPA、およびFlywayが含まれるが、編集のたびに
データベースへ確定する方式は、手動保存と変更破棄の操作モデルに追加の作業領域を必要とする。

## 決定

- プロジェクトをフォルダ単位で管理する。
- 構造化データをプロジェクトフォルダ直下の`project.json`へ保存する。
- `project.json`はUTF-8の整形済みJSONとし、内容を閲覧可能にする。ただし、利用者による
  手動編集はサポート対象外とし、読み込み時に構造とデータ整合性を検証する。
- 地図背景と添付ファイルをプロジェクトフォルダ内のアセットファイルとして保持する。
- 編集中の状態をJavaバックエンドのメモリ上で管理し、手動保存時にJSONへ永続化する。
- 自動復旧には、正式保存と分離したJSONスナップショットを使用する。
- H2、Spring Data JPA、およびFlywayは採用しない。
- JSONにスキーマバージョンを保持し、読み込み時の検証と将来の形式移行に使用する。
- スキーマバージョンは整数の`1`から開始し、非互換変更時に1ずつ増やす。プロジェクト、
  カタログ、および復旧形式は別々に管理し、各版のJSON Schemaと隣接バージョン間の
  変換処理をリポジトリへ保持する。
- プロジェクト、背景、エリア、墓所、人物、およびアセットをトップレベルで独立して保持し、
  データ間を変更されないUUIDで関連付ける。同じデータを複数箇所へ重複して埋め込まない。
- 永続データのUUIDはJavaバックエンドがVersion 4で生成する。コピー時は新しく発行し、
  インポート時は元UUIDを維持する。重複時は対象データを変更せず処理を拒否する。
- 表示順が必要なデータは配列順へ暗黙に依存せず、明示的な順序値を保持する。
- 対応可能な古いスキーマは新しいアプリで自動変換し、変換前にプロジェクト一式を退避する。
- 変換前退避は`backup/schema-migration/`の日時付きZIPとし、`project.json`と`assets`を
  含める。既存の`backup`は含めず、ZIPの作成と検証に失敗した場合は変換を開始しない。
- 変換前退避は、1日1回・3世代の通常自動バックアップとは別に管理する。
- 変換前退避も最新3世代を保持し、新しいZIPの検証成功後にだけ古い世代を削除する。
  作成または検証に失敗した場合は、既存の退避データを維持する。
- アプリより新しいスキーマは更新せずに拒否し、変換失敗時も元データを変更しない。
- 正式保存では一時ファイルへの書き出しと検証を完了してから、`project.json`を切り替える。
- 新規アセットはJSONより先に正式領域へコピーして検証し、旧アセットはJSON切り替え後に
  削除する。JSONの切り替えを保存の確定点とし、途中失敗時は旧データを維持する。
- JSONから参照されないアプリ管理下のアセットは、次回起動時に検出して清掃する。
- アセットの正式名はUUIDと許可済み拡張子で構成し、元ファイル名はJSONメタデータに保持する。
  未参照のUUID形式ファイルだけを自動清掃し、その他の不明なファイルは削除せず警告する。
- アセットメタデータにはUUID、元ファイル名、相対パス、MIMEタイプ、サイズ、SHA-256、
  および追加日時を保持し、読み込みとデータ移送時の破損・差し替え検出に使用する。

## 理由

- 手動保存、変更破棄、および自動復旧の責務を明確に分けられる。
- プロジェクトフォルダを単位として移動、コピー、バックアップ、および復元を行える。
- 5,000墓所規模では、メモリ上での編集と検索が現実的である。
- データ形式を確認しやすく、データベース専用ツールを必要としない。
- データベースとアセットファイルの別個のバックアップ処理を避けられる。

## 影響

- 一意制約、参照整合性、および業務上の不変条件はJavaバックエンドが検証する。
- JSON読み込み時にUUIDの重複と参照切れを検出する必要がある。
- 保存失敗やファイル破損に備え、全体書き出し、検証、安全な置換、およびバックアップが必要になる。
- JSONスキーマの変更に対するバージョン変換処理を実装し、テストする必要がある。
- スキーマ変換前の退避に必要な空き容量を確認し、退避失敗時は変換を開始しない必要がある。
- アセット参照にはプロジェクトルートからの相対参照を使用し、パストラバーサルを防止する必要がある。

## 採用しなかった選択肢

- H2、Spring Data JPA、およびFlyway: トランザクション、一意制約、外部キー、および
  SQL検索を利用できる一方、手動保存と変更破棄のために別の編集状態を管理する必要があり、
  MVPの規模では永続化構成が複雑になる。
- 単一アーカイブファイルへの直接保存: 持ち運びやすい一方、通常編集時のアセット追加や
  安全な差し替えが複雑になる。単一ファイル化は`.hakamap`エクスポート形式として使用する。

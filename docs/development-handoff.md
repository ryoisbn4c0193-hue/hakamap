# 開発引き継ぎメモ

最終更新日: 2026-07-22

## 現在の目的

Hakamap初期版の機能要件を、利用者への質問を通じて確定し、`docs/requirements`配下へ
随時反映している。要件定義全体の進捗は約60%と見積もっている。

## 今回完了した作業

- 対象利用者を寺院の墓地管理者とし、紙の墓地図と埋葬記録のデジタル化を主目的とした。
- 画像またはPDFの背景上へ墓所を配置し、人物情報を画面から入力する初期フローを定めた。
- 墓所、人物、エリア、管理番号、組名、拡張項目、マスター管理の要件を定めた。
- 墓所の内部主キーをUUID、業務キーを「エリア＋管理番号」とした。
- エリアを1エリアにつき1多角形とし、エリア同士の重なりを禁止した。
- 墓所・人物・添付ファイルのごみ箱、復元、完全削除の基本規則を定めた。
- 標準項目と拡張項目の検索、範囲検索、条件結合、文字列正規化の要件を定めた。
- 拡張項目の型、適用先、変更、利用停止、削除、およびプロジェクト間コピーを定めた。
- 墓所と人物へPNG、JPEG、WebP、PDFを添付する要件を定めた。
- 添付ファイルの容量、プレビュー、PDFリンク、および削除・復元の要件を定めた。
- PDF内の確認付き外部Webリンクを許可する決定に合わせ、オフライン動作設計を更新した。
- 墓所をピンではなく長方形で表現する決定に合わせ、`AGENTS.md`を更新した。

## 次に行うこと

ユーザーは外出先でスマートフォン版ChatGPTにGitHubの内容を参照させ、要件定義を
継続する予定である。次回、そこで検討した内容がテキストで連携された場合は、現在の
要件文書とGit差分を確認し、確定事項を該当文書へ反映する。矛盾がある場合は勝手に
上書きせず、ユーザーへ確認する。

地図編集機能の要件定義から再開する。次の質問は、墓所や建物などに対して初期版で
以下の操作を提供するか、特に複数選択と一括移動を必要とするかである。

- クリックによる選択
- ドラッグによる移動
- ハンドルによるサイズ変更と回転
- 矢印キーによる微調整
- 複製と削除
- 複数選択、一括移動、一括削除
- UndoとRedo

墓所の複製では位置と外観だけをコピーし、新しい「エリア＋管理番号」を必須とし、
人物や添付ファイルはコピーしない案を提示しているが、まだ確定していない。

その後、次の未決領域を順次確認する。

- 地図要素の色、線、ラベル、レイヤー、スナップ、ズーム、パン
- 地図背景の容量および解像度上限
- プロジェクトの作成、名称変更、削除、ごみ箱
- 自動保存と入力途中の扱い
- 画面構成と画面遷移
- インポート、エクスポート、印刷、帳票
- 想定データ件数、性能目標、対応Windows、受け入れ条件

## 注意事項

- 外部のChatGPTがGitHub上の最新情報を参照できるよう、PCでの作業終了時には変更を
  コミットしてリモートへプッシュする運用とする。ただし、コミットとプッシュは
  ユーザーから指示を受けた後に実行する。
- `.devcontainer/devcontainer.json`には今回の要件定義開始前から未コミット変更があり、
  内容を変更していない。
- 要件への回答を受け取るたびに文書を更新する運用としている。更新による応答遅延が
  問題になった場合は、区切りごとの一括更新へ変更する可能性がある。

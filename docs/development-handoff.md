# 開発引き継ぎメモ

最終更新日: 2026-07-23

## 現在の目的

レビュー合格済みの要件・基本設計・詳細設計に従い、Hakamap MVPを
`docs/implementation-plan.md`のPhase単位で実装する。Phase 0のビルド・依存・品質基盤と
Phase 2のJSON保存モデルとRepositoryまで完了し、次はPhase 3の保存トランザクションと
履歴を実装する。
- 当時のコミット`aa92223`に対する再レビューは合格となり、重要な矛盾が解消され、
  MVP、将来拡張、および未決事項の境界が明確であると確認された。
- 基本アーキテクチャとして、React、TypeScript、PixiJS、Java 21、Spring Boot、および
  既定ブラウザによる構成を採用した。Tauri、Electron、およびJavaFXは採用しない。
- 永続化はプロジェクトフォルダ内のJSONとアセットファイルを採用し、H2、Spring Data JPA、
  およびFlywayは採用しない。

- MVPでは手動バックアップを設けず任意コピーには`.hakamap`エクスポートを使い、

- 破損等は復元候補から除外して通知するが自動削除せず、MVPでは手動削除を設けず、

- 上書き復元前に`backup/pre-restore/pre-restore-<UTC日時>.zip`へ現在の保存済み状態を必ず退避し、

- 復元前退避も別グループから復元可能とし、復元後はバックアップ内のプロジェクト情報を使い、
  保存場所・デフォルト設定は維持、履歴は破棄、リビジョンは0として再読み込みすることを定めた。
- ズームを10～800%、1操作約20%とし、ホイール・ピンチ・ボタン・`+`／`-`／`0`で操作し、
  背景・エリア・墓所を対象とする全体表示と初回表示を定めた。
- パンは中ボタンドラッグまたはSpace＋左ドラッグとし、範囲制限なし、選択や保存座標を変えず、
  テキスト入力中はキーボードショートカットを無効にすることを定めた。
- Shiftは選択追加に使わず、矢印の10単位移動、ドラッグ移動の水平・垂直制約、サイズ変更と
  背景拡縮の縦横比維持に使用し、ドラッグ途中の押下・解除にも追従することを定めた。
- エリア色を青、緑、オレンジ、紫、赤、グレーの固定IDとし、初期青、境界線2px相当、
  同色20%塗りつぶし、選択時は別の強調線、色だけに依存しない表示とすることを定めた。
- MVPでは任意色、透明度、線幅変更を設けず、不明IDを拒否し、将来の表示色調整でも
  保存済みIDを維持することを定めた。
- PNG・JPEG・WebP背景は元画像だけ、PDFは変換後PNGだけを保存し、二重保存しないことを定めた。
- 背景透明度を100%固定、実寸縮尺とアプリ内縮小をMVP対象外とし、上限超過時は理由と
  許容上限を示して拒否し、これらを将来拡張候補へ整理した。
- Project、Catalog、RecoveryのSchema v1をMVP初期形式とし旧版変換は持たず、v2以降の追加時に
  Schemaと隣接版Migratorを同時追加することを定めた。
- 解消済みの空の未決事項見出しを削除し、手動バックアップをMVP外の将来候補へ整理した。
- 要件、基本設計、ADR、READMEを横断確認し、MVPと将来拡張の境界、未決事項、文書間の
  整合性を確認して基本設計完了と判定した。
- 基本設計レビューで、背景描画上限、所属エリア判定、色の一意性、NAS排他、
  自動バックアップ契機の5点が修正必須として指摘された。
- 高解像度背景は1024px以下のタイル階層を端末キャッシュへ生成し、表示範囲周辺だけを
  PixiJSへ読み込む方式とした。原本だけをプロジェクトへ保存し、タイルはデータ移送へ含めない。
- GPU最大テクスチャサイズを確認し、最大背景と1GB目標を最低環境のWindows実機で試験し、
  満たせない場合はリリース前に上限または目標を見直すことを定めた。
- 墓所の中心点を含むエリアへ所属させ、共有境界上では`displayOrder`最小を使用し、候補なしは
  未割当、墓所全体が収まらない場合は保存可能な警告とすることを定めた。
- 所属と警告を各地図操作・Undo／Redo・エリア変更時に再計算し、現在の所属UUIDを業務キーへ
  使用するがJSONへは保存しないことを定めた。
- エリア色を状態色と重複しない12プリセットへ拡張し、プロジェクト内で一意、エリア上限12件、
  未使用色の自動設定、削除後の色再利用、使用中選択肢の無効化を定めた。
- 選択・警告・エラー・無効の状態色を固定して前面表示し、将来の色追加では既存IDを維持して
  Schema、検証、UI、エリア上限を同時更新することを定めた。
- ネットワーク上の保存先は、機能検査に合格したWindows 11のSMB 3.x共有に限定し、
  単一端末・単一Windowsユーザーで利用することを定めた。
- 新規作成・再関連付け時に読み書き、SHA-256、削除、別Javaプロセスとのロック競合、
  原子的置換を検査し、失敗時は編集を拒否して非原子的保存へフォールバックしないことを定めた。
- SMB共有では、別端末とのロック競合、LAN切断、共有サーバー再起動、保存中の切断、
  再接続後の整合性をWindows 11実機で試験することを定めた。
- 自動バックアップはデータ変更を含む手動保存成功直後に判定し、Windowsのローカル日付で
  1日1回、新規プロジェクトでは最初の保存後に作成することを定めた。
- バックアップ失敗時も手動保存は成功扱いとし、警告して次の手動保存成功後に再試行する。
  起動・定時・終了時には開始せず、更新日時、リビジョン、Undo／Redo履歴を変更しない。
- ブラウザを閉じてもバックエンドは終了させず、デスクトップまたはスタートメニューの
  ショートカット再実行で既存バックエンドの画面を開き直すことを定めた。
- 画面内に「Hakamapを終了」とヘルプを設け、初回起動時にも終了・再表示方法を案内する。
  MVPではタスクトレイへ常駐しない。
- エリア名と管理番号は入力表記を保存しつつ、NFKC、英字大小、Unicode空白、
  ひらがな・カタカナを統一した導出キーで一意性を判定することを定めた。
- 座標と寸法は小数第3位へ丸めた候補値で幾何判定し、丸め後の寸法0以下または重なりは
  コマンド全体を拒否することを定めた。
- 墓所の完成状態と警告を分離し、未割当・未採番は複数理由を持つ情報未完成、
  エリアからのはみ出しは完成状態へ影響しない保存可能な警告とすることを定めた。
- エリアの非表示は所属と業務キーへ影響させず、削除とUndo／Redo時は残存する候補状態から
  所属、完成状態、および警告を再計算することを定めた。
- 墓所コピーは幅・高さ・内部回転角度だけを引き継ぎ、管理番号、墓所名、備考、人物、
  添付を持たない新規UUIDの墓所を作ることを定めた。
- 一括コピーは元の位置関係と同じ移動量を使用して全件成功または全件失敗とし、
  コピー元のデータと更新日時を変更しないことを定めた。
- 単体墓所はドラッグ、一括作成は行列・寸法・間隔・左上位置、範囲埋めは長方形範囲と
  寸法・間隔を入力し、左から右、上から下へ生成することを定めた。
- 範囲へ完全に収まる墓所だけを作り、生成墓所は新規UUID、回転0、同一更新日時、
  その他情報なしとし、重なり等は全件拒否、未割当・はみ出しは警告許可と定めた。
- 一括採番は接頭辞、開始番号、1～10桁、接尾辞、増分1で生成し、視覚的な行を左から右、
  上から下へ並べて割当予定をプレビューすることを定めた。
- 採番済み・未割当墓所を含む場合や文字数・業務キー違反時は全件拒否し、複数エリアでは
  番号を継続し、プレビューの手動並べ替えはMVP外と定めた。
- JSON Schema検証にはJackson 3・Java 17以上・Draft 2020-12対応の
  `com.networknt:json-schema-validator:3.0.6`を固定採用し、外部Schema取得を無効にして
  Apache License 2.0の表示を配布物へ含めることを定めた。
- ドラッグは確定まで、フォームは保存・適用、矢印キー連続入力はキーを離すまでを
  1履歴とし、実質的な変更がない操作は履歴・リビジョン・日時を変更しないことを定めた。
- 未保存判定は決定的な保存モデルの状態フィンガープリントで行い、Redoは元コマンドを
  再実行せず最初に確定したUUID・日時を含む変更後差分を再適用することを定めた。
- 未割当・はみ出し警告が新しく発生する場合だけ反映前に確認し、未採番、既存警告の継続、
  Undo／Redo、保存では再確認しないことを定めた。
- 一括警告は理由別件数をまとめ、候補をサーバーのメモリへ5分間だけ保持する単一使用トークンで
  同一リビジョンに対して確定し、キャンセル・期限切れ・競合時は反映しないことを定めた。
- 変更なしの保存はJSON・日時・リビジョン・履歴・バックアップを変更しないno-opとし、
  保存失敗時は旧正式データと未保存編集を維持して再試行・別保存・キャンセルを提示する。
- Project保存は検証済み一時JSONの原子的置換を確定点とし、確定時だけ更新日時を反映する。
  確定後の不要ファイル清掃失敗は保存成功を維持して警告し、次回起動・保存で再試行する。
- 自動バックアップは手動保存確定後の別トランザクションとし、失敗しても保存成功と
  既存3世代を維持することを定めた。
- 自動復旧候補は保存日時と復旧日時を表示し、適用時は正式ファイルを上書きせず、
  未保存状態、revision 0、空の履歴として開くことを定めた。
- 復旧後は手動保存または変更破棄を選べ、Recovery・Project・一時アセットの全検証に
  成功しない限り部分復旧せず、正式保存データを維持することを定めた。
- バックアップ復元は検証済みの復元前退避を必須とし、不変UUIDの必要アセットを先に配置して
  `project.json`の原子的置換を確定点とすることを定めた。
- 原子的置換中の切断で結果を検証できない場合は自動再試行せず、再接続後に新旧JSONの
  SHA-256で確定状態を判定し、どちらでもなければ元保存先へ上書きしないことを定めた。
- APIを`/api/v1/projects/{projectId}`境界とし、初回は全スナップショット、変更成功時は
  確定差分・導出状態・最新revisionだけを返すことを定めた。
- 長時間処理はセッションに紐づくOperation IDを返し、約500ミリ秒間隔で進捗取得、
  確定段階後はキャンセル不可、未取得結果は10分で破棄することを定めた。
- 一括Problem Detailsは個人情報を含まない違反を最大100件と総件数で返し、
  API v1と保存Schemaのバージョンを独立管理することを定めた。
- 検索語は最大200文字、結果は総件数付きの不透明カーソルで200件ずつ取得し、
  revision変化時は先頭から再検索することを定めた。
- API JSON 10MB、ネスト50、文字列1MB、数値64文字、Project 100MB、Catalog 10MB、
  Recovery 110MBのパーサー防御値を定め、MVPの画像配信ではRangeを使用しない。
- フロントエンドではJavaの確定状態をTanStack Query、選択やviewportなどのUI状態を
  Zustand、入力途中と地図プレビューを一時状態として分離することを定めた。
- 未確定フォームがある選択変更では破棄またはキャンセルを確認し、リビジョン競合時は
  地図操作を自動再実行せず、フォーム入力だけを確認可能な状態で保持することを定めた。
- PixiJSの描画順を背景、エリア、墓所、操作オーバーレイとし、通常時は墓所を優先して
  エリアを専用編集モードで操作することを定めた。
- 非表示エリアを描画・選択対象外とし、矩形選択では辺・角の接触を含めて墓所と
  少しでも交差すれば選択することを定めた。
- スナップ検出距離をズームに依存しない画面上8pxとし、最小補正量、角、辺、墓所、
  エリア、表示順、UUIDの順で候補を一意に決めることを定めた。
- 一括採番では高さが小さい墓所を基準に上下方向が50%以上重なる墓所を同じ視覚的な行とし、
  行内を中心X、行を中心Y、最終同値をUUIDで安定して並べることを定めた。
- テストデータは架空データと生成fixtureだけを使用し、小規模10墓所、標準1,000墓所、
  上限5,000墓所のデータ規模を目的別に使用することを定めた。
- 日常検査では単体・結合と標準規模までを扱い、上限データ、最大背景、NAS障害、
  性能、およびパッケージをWindows重負荷検査として分離することを定めた。
- 詳細設計レビューは完成度約90%の条件付きOKとなり、実装開始前の修正必須4点として
  人物取得方式、Operation API、ファイル選択API、およびCSRF取得方式が指摘された。
- 初回スナップショットから人物全件を除外し、選択墓所の人物を100件ずつ取得する専用API、
  仮想スクロール、人物変更差分、および墓所単位キャッシュを定めた。
- 長時間処理をProject配下の保存・エクスポート・バックアップ復元、Catalog配下の
  インポート、ごみ箱Projectの復元へ分離し、DTO、結果、キャンセル段階を定めた。
- ファイル選択開始APIへ単一・複数・フォルダ、用途列挙、キャンセル、多重起動、

- 端末管理データは対象確認後に削除してアプリを終了し、次回は空の一覧で起動すること、
  プロジェクト本体はごみ箱移動と完全削除だけで削除することを定めた。
- ログはUTF-8、1ファイル10MB、現行＋過去5世代、最大約60MBとし、古い世代から
  削除することを定めた。
- ログには個人情報、絶対パス、セッション情報、編集履歴を記録せず、ログ書き込み失敗だけを
  理由に編集データを失わないことを定めた。
- プロジェクト名は必須・最大50文字・前後の半角全角空白を除去し、空文字、改行、制御文字を
  拒否して同名を許可し、UUIDで識別することを定めた。
- 同名は更新日時と保存場所末尾で区別し、プロジェクト名を内部ファイル・フォルダ名へ直接
  使用せず、絶対パス全体をブラウザへ返さないことを定めた。
- 新規プロジェクトは選択した親フォルダ直下の`hakamap-project-<project-uuid>`へ作成し、
  表示名変更ではフォルダ名を変更せず、既存フォルダを上書きしないことを定めた。
- 手動移動・改名後は再関連付け可能とし、ごみ箱内のフォルダ名にはUUIDだけを使用することを定めた。
- 背景は100MB・1億画素・各辺30,000px、添付は25MB・4,000万画素・各辺15,000pxを
  上限とし、プロジェクト全体は固定上限なしとすることを定めた。
- 上限超過時は自動縮小せず拒否し、拡張子、MIMEタイプ、実内容、空き容量を検証することを定めた。
- 添付サムネイルは240×180px枠、縦横比維持・拡大なし、WebP品質80とし、アセットSHA-256を
  基準に端末キャッシュへ保存することを定めた。
- サムネイルキャッシュはプロジェクト、バックアップ、エクスポートに含めず、端末管理データ
  削除の対象に含めることを定めた。
- サムネイルキャッシュは最大500MBとし、表示中を除く最終利用の古いものから削除し、
  元アセットがないものは起動時の清掃候補とすることを定めた。
- 生成失敗時は汎用アイコンを表示し、設定画面からキャッシュだけを削除でき、必要時に
  自動再生成することを定めた。
- 文字数上限はエリア名25、管理番号25、墓所名50、墓所備考1,000、添付表示名100、
  添付説明500、元ファイル名255文字とすることを定めた。
- 文字数はUnicodeコードポイント単位で数え、上限超過を切り捨てず、制御文字を拒否し、
  墓所備考と添付説明だけ改行を許可することを定めた。
- 人物のMVP項目は氏名、戒名、登録日時、更新日時とし、氏名か戒名の一方を必須、各50文字、
  重複可、前後空白除去、改行・制御文字不可とすることを定めた。
- 人物変更時は人物と所属墓所の更新日時を更新し、組名は墓所単位の将来拡張とすることを定めた。
- MVP検索は単一キーワードでエリア名、管理番号、墓所名、人物の氏名・戒名、添付表示名・
  説明を部分一致検索し、墓所備考とファイル内容は対象外とすることを定めた。
- 検索時は英字大小、全半角、Unicode空白、ひらがな・カタカナを同一視し、墓所UUIDで
  結果を重複排除することを定めた。
- 数値・日付範囲、項目別条件、AND／OR指定は将来拡張とした。
- 検索結果はエリア表示順、管理番号の自然順、墓所名、UUIDで安定ソートし、未割当は
  全エリアの後、未採番は同一エリアの採番済み墓所の後に表示することを定めた。
- 検索結果の並びは保存データを変更せず、MVPでは利用者による切り替えを設けないことを定めた。
- エリア表示順は作成順で末尾追加、名称変更では維持、削除時は詰め、Undo時は元位置へ戻し、
  インポート・エクスポート・バックアップ・復元で維持することを定めた。
- エリアの手動並べ替えはMVP対象外とし、将来拡張候補へ追加した。
- 性能基準は5,000墓所・20,000人（平均4人）とし、人物数に登録を禁止する固定上限を
  設けないことを定めた。
- 人物一覧は仮想スクロールを使い、検索データはJava側で墓所単位に集約して、人物全件を
  Reactへ常時送信しないことを定めた。
- 人物表示順は墓所内の登録順で末尾追加、編集では維持、削除時は詰め、Undo時は元位置へ戻し、
  インポート・エクスポート・バックアップ・復元で維持することを定めた。
- 人物の手動並べ替えはMVP対象外とし、将来拡張候補へ追加した。
- 添付は新規追加時に末尾へ配置し、ドラッグ＆ドロップまたは「上へ」「下へ」で並べ替え、
  並べ替えをUndo／Redo対象とすることを定めた。
- 複数追加時は選択ダイアログの順序を使い、削除時は詰め、Undo時は元位置へ戻し、
  `displayOrder`をインポート・エクスポート・バックアップ・復元で維持することを定めた。
- エクスポートは`.hakamap`拡張子のZIPとし、マニフェスト、プロジェクトJSON、アセットを含め、
  バックアップ、ロック、復旧、一時、キャッシュ、ログを除外することを定めた。
- マニフェストへ形式・アプリバージョン、作成日時、プロジェクトUUID、各ファイルのパス・
  サイズ・SHA-256を保持し、インポート時にZIPパスと全整合性を検証することを定めた。
- 未保存変更がある場合は保存成功後だけエクスポートし、古い保存状態だけの出力は設けず、
  保存失敗または切断時は開始しないことを定めた。
- エクスポート自体は更新日時、編集リビジョン、Undo／Redo履歴を変更せず、成功後も
  現在のプロジェクトを開いた状態にすることを定めた。
- インポート前は現在のプロジェクトの未保存変更を保存または破棄して閉じ、ロックと履歴を
  解放し、通常インポートではプロジェクトと配下のUUID・名称・日時を維持することを定めた。
- 同一UUIDは拒否または別プロジェクト化し、成功後は取り込み先を開き、失敗時は不完全な
  フォルダを残さず一覧へ戻り、デフォルト設定を変更しないことを定めた。
- インポートは選択先の`.hakamap-import-<random-uuid>.tmp`へストリーミング展開し、絶対・
  親参照パス、リンク、パス重複、未宣言ファイル、宣言サイズ超過を拒否することを定めた。
- 全検証後の正式名への変更を確定点とし、成功後だけカタログ登録し、削除できない残骸は
  隔離して次回起動時にも清掃することを定めた。
- インポート防御上限はマニフェスト10MB、プロジェクトJSON 100MB、100,010ファイル、
  圧縮率1,000倍、展開後合計＋500MBの空き容量とすることを定めた。
- 展開中に宣言サイズを超えた場合は中止し、ZIPを再帰展開せず、防御上限は通常編集の
  プロジェクト全体容量には適用しないことを定めた。
- 100,010ファイルは性能保証ではなく限界試験とし、通常性能試験は5,000墓所、平均2添付、
  約10,000添付を代表値として、ウイルス対策有効なWindows 11実機で所要時間を測定する。
- Windows 11標準のEdgeを基準とし、Edge、Chrome、Firefoxのリリース時最新安定版と
  1世代前を動作保証対象とすることを定めた。
- その他の既定ブラウザも起動を試み、必須機能が不足する場合は対応ブラウザを案内し、
  ブラウザ固有のファイルシステムAPIへ依存しないことを定めた。
- 人物単独のCSV／Excel一括入出力はMVP対象外とし、画面から1件ずつ管理し、プロジェクト
  全体の`.hakamap`には人物を含めることを定めた。
- CSVテンプレート、文字コード、検証エラー、重複判定、追加・更新ルールは将来拡張へ移した。
- 印刷・PDFはA4／A3、縦横選択、初期A4横、余白10mm固定とし、現在範囲または選択エリアを
  縦横比維持で用紙内へ自動縮尺することを定めた。
- 実寸縮尺、ヘッダー、フッター、ページ番号はMVP対象外とし、印刷とPDFで共通レイアウトを
  使用して出力前にプレビューすることを定めた。
- PDFは選択した用紙設定を反映した1ページ、画像は同じ用紙設定を反映した白背景のPNG 1枚とし、
  300 DPI相当の固定解像度で共通プレビューから出力することを定めた。
- Windowsの製品名と発行元を`Hakamap`、説明を「墓地マップ管理アプリ」、バージョンを
  3要素形式、インストーラー名を`Hakamap-<バージョン>.exe`とすることを定めた。
- アプリとインストーラーで16～256ピクセルを含む同じ`.ico`を使用し、正式版まで仮アイコンを
  差し替え可能にすること、およびアップグレード用製品識別子を固定することを定めた。
- 入力項目とファイル容量の上限を未確定へ変更した。
- 添付PDFを1ページ限定とし、複数ページを拒否して画像としてプレビューする仕様へ変更した。
- 管理番号を墓所の完成条件とし、未採番および情報未完成の墓所を保存可能とした。
- 通路、入口、建物、および自由地図要素をMVP対象外へ変更した。
- 拡張項目、マスターデータ、およびプロジェクト間設定コピーを将来候補へ移動した。
- 詳細検索仕様と人物入力項目を未決または将来候補へ移動した。

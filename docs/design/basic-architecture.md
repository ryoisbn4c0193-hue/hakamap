# 基本アーキテクチャ設計

## 目的

HakamapのMVPを構成するアプリケーション境界、採用技術、および主要コンポーネントの
責務を定める。

## 採用構成

- フロントエンドはReactおよびTypeScriptで実装する。
- 地図描画と地図上の対話操作にはPixiJSを使用する。
- バックエンドと業務ロジックはJava 21で実装する。
- ローカルAPIとフロントエンド配信にはSpring Bootを使用する。
- 利用者の既定ブラウザでUIを表示する。
- バックエンドは`127.0.0.1`だけで待ち受ける。
- 本番UIとAPIはSpring Bootから同一オリジンで配信する。
- Tauri、Electron、およびJavaFXは採用しない。
- Windows配布には`jpackage`によるユーザー単位の`.exe`インストーラーを使用し、`jlink`で
  最小化したJava 21ランタイムを同梱する。

## コンポーネントの責務

### Reactフロントエンド

- 画面構成、フォーム、サイドバー、およびプロパティパネルを表示する。
- PixiJSアダプターを介して地図を描画する。
- ポインター、キーボード、選択、および地図編集操作を受け付ける。
- APIのDTOを画面用状態へ変換する。
- 業務上の不変条件や永続化処理を重複実装しない。

### Javaバックエンド

- 墓所、エリア、人物、添付、プロジェクトなどの業務ルールを管理する。
- プロジェクトを開いている間の編集状態を管理する。
- 手動保存、読み込み、インポート、エクスポート、バックアップ、および自動復旧を処理する。
- 利用者が指定するプロジェクト保存先へのファイル操作を行う。
- 入力検証、整合性検証、およびエラー分類を行う。
- 開いているプロジェクトを`ProjectAggregate`としてメモリ上に保持し、編集状態の正とする。
- データ変更をドメインコマンドとして実行し、Undo／Redoとリビジョンを管理する。
- Undo／Redo履歴にはプロジェクト全体のコピーではなく、操作前後の差分を保持する。

### PixiJSアダプター

- PixiJS固有オブジェクトとReactコンポーネントの境界を提供する。
- 地図座標と画面座標の変換を行う。
- ズーム、パン、選択表示、ガイド、およびスナップ表示を扱う。
- 保存する地図座標を画面ピクセルへ置き換えない。

## 編集状態と保存

- 同時に開くプロジェクトは1件だけとする。
- 開いているプロジェクトは`.hakamap.lock`で排他的にロックし、同一プロジェクトの同時編集を拒否する。
- 保存先の切断またはロック喪失時は編集状態をローカル復旧領域へ保護し、元保存先への
  編集と保存を停止する。
- 開いているプロジェクトの編集状態はJavaバックエンドの`ProjectAggregate`が管理する。
- UI操作はローカルAPIを介して編集状態へ反映する。
- プロジェクトを開くたびに編集セッションの`revision`を0から開始する。
- データ変更、Undo、およびRedoが成功するたびに`revision`を1増やす。
- データ変更要求にはクライアントが認識する`expectedRevision`を付け、一致する場合だけ処理する。
- リビジョンが一致しない場合は編集状態を変更せず競合エラーを返し、UIに最新状態を再取得させる。
- ズーム、パン、選択、および検索など、保存対象外の操作では`revision`を増やさない。
- 通常のプロジェクト保存は利用者が明示的に実行する。
- 未保存変更の有無をバックエンドで追跡する。
- 自動復旧用スナップショットは通常保存およびバックアップと分離する。
- プロジェクトはフォルダ単位で管理し、構造化データは`project.json`へ保存する。
- 地図背景と添付ファイルはプロジェクトフォルダ内のアセットとして保持する。
- 編集中の状態はJavaバックエンドのメモリ上で保持し、H2、Spring Data JPA、および
  Flywayは使用しない。
- 自動復旧用スナップショットには、正式保存とは分離したJSONを使用する。

## ローカルファイル操作

- ブラウザへプロジェクト保存先の絶対パスを公開しない。
- 任意フォルダの選択が必要な操作では、Java側からWindowsのネイティブファイル選択
  ダイアログに相当する補助機能を提供する。
- MVPではJava標準のSwing `JFileChooser`をEDT上で表示し、ファイル選択とフォルダ選択に対応する。
- 選択した絶対パスはJava側の一時コンテキストへ保持し、Reactには選択結果を識別するIDだけを返す。
- 選択機能をインターフェースで分離し、将来JNA経由のWindows `IFileDialog`へ差し替え可能にする。
- 選択されたパスはバックエンド内で検証し、利用者が許可した操作にだけ使用する。

## 採用しない構成

- ElectronによるChromiumとNode.jsの同梱
- TauriによるRustバックエンドの追加
- JavaFXによるUIの再実装
- UIとAPIを別オリジンで公開し、制限のないCORSで接続する構成
- LAN上の他端末へローカルAPIを公開する構成

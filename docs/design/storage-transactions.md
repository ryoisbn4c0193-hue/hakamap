# 保存・バックアップ・復旧トランザクション詳細設計

## 目的

Project、アセット、Catalog、Recovery、およびバックアップを、強制終了、ストレージ切断、
ロック喪失、容量不足、検証失敗が発生しても、最後に正常確定した状態または検証可能な
未保存復旧状態から再開できる順序で更新する。

## 確定点

| 処理 | 確定点 |
| --- | --- |
| Project手動保存 | 検証済み一時JSONの`project.json`への原子的置換成功 |
| Catalog更新 | 検証済み一時JSONの`catalog.json`への原子的置換成功 |
| Recovery更新 | 検証済み一時JSONの`<project-id>.recovery.json`への原子的置換成功 |
| 自動バックアップ | 検証済み一時ZIPの正式`.zip`名への原子的変更成功 |
| エクスポート | 検証済み一時ファイルの利用者指定`.hakamap`への原子的変更成功 |

- 確定点より前の失敗では正式状態を変更しない。
- 確定点より後の清掃失敗では確定済み状態を巻き戻さず、安全な再試行対象として記録する。
- 非原子的な上書きを確定処理の代替として使用しない。

## 排他と実行制御

- `ProjectStorageTransactionCoordinator`が、開いているProjectごとに保存系処理を直列化する。
- 手動保存、復元、インポート、プロジェクト移動、および別保存を同時に実行しない。
- 自動バックアップとエクスポートは、同じProjectの手動保存または正式ファイル切り替えと
  同時に実行しない。
- 処理開始前に編集セッションの`expectedRevision`、`.hakamap.lock`、保存先接続、
  基準`project.json`のSHA-256、および必要空き容量を確認する。
- 長時間処理中は新しいデータ変更、プロジェクト切り替え、および終了を受け付けない。
- 操作直前検査に失敗した場合はファイル作成を開始せず、編集停止または競合状態へ移行する。

## 変更なしの手動保存

- 現在の状態フィンガープリントが`lastSavedFingerprint`と一致する場合は、成功したno-opとして返す。
- `project.json`、アセット、Recovery、Catalog、およびバックアップを変更しない。
- `project.updatedAt`、編集`revision`、Undo／Redo履歴、および保存位置を変更しない。
- 画面にはエラーを表示せず、「変更はありません」と通知できる結果を返す。

## Project手動保存

### ステージ1: 事前検査

1. 保存系処理の排他権を取得する。
2. `expectedRevision`と現在の編集`revision`を照合する。
3. 未保存変更の有無を状態フィンガープリントで確認する。
4. Projectロックと保存先の書き込み可否を確認する。
5. ディスク上の`project.json`のSHA-256と基準値を照合する。
6. ステージング済みアセット、既存アセット、および必要空き容量を検査する。

- いずれかに失敗した場合は正式領域へ書き込まない。
- 外部変更を検出した場合は元保存先へ上書きせず、基本設計の別保存または破棄へ移行する。

### ステージ2: 新規アセットの正式配置

1. 各ステージングアセットを再度読み、サイズ、SHA-256、MIMEタイプ、および画像制約を確認する。
2. Project内の正式相対パスと同じディレクトリへ、一意な一時名でコピーする。
3. 一時ファイルを`FileChannel.force(true)`で同期する。
4. 一時ファイルをUUIDの正式ファイル名へ原子的に移動する。
5. 正式配置後のファイルをサイズとSHA-256で再確認する。

- UUIDの正式名が既に存在する場合は内容を上書きせず、UUID衝突または残存ファイルとして保存を中止する。
- この段階で失敗した場合は新規に作った一時ファイルと未参照の正式ファイルを清掃対象とし、
  旧`project.json`と旧アセットを維持する。

### ステージ3: JSON候補の作成と検証

1. 保存確定用のUTCミリ秒時刻を発行する。
2. Project候補の`project.updatedAt`だけを保存確定時刻へ置き換える。
3. 決定的な順序でProject保存モデルを構築する。
4. Project全体の業務不変条件を検証する。
5. Projectルートと同じディレクトリの一意な一時ファイルへUTF-8 JSONを出力する。
6. 一時JSONを`FileChannel.force(true)`で同期する。
7. 出力済みJSONを再読込し、Project Schema、Java保存モデル、および業務不変条件を再検証する。
8. 一時JSONのSHA-256と状態フィンガープリントを算出する。

- 一時ファイル名は`.project-<random-uuid>.tmp`とし、正式なProjectとして読み込まない。
- 失敗時は一時JSONを削除し、メモリ上の`project.updatedAt`を変更しない。

### ステージ4: JSONの確定

1. 確定直前にProjectロック、接続、基準JSONのSHA-256を再確認する。
2. `Files.move`へ`ATOMIC_MOVE`と`REPLACE_EXISTING`を指定し、一時JSONを`project.json`へ置換する。
3. 置換後の`project.json`を再読込し、期待したSHA-256と一致することを確認する。

- 原子的置換の成功をProject手動保存の確定点とする。
- 置換に失敗した場合は旧`project.json`を正式状態とし、編集状態とRecoveryを維持する。
- 置換成功後の再読込検証に失敗した場合は致命的な保存整合性エラーとして編集を停止し、
  メモリとRecoveryを維持して元保存先へ追加書込みしない。

### ステージ5: メモリ状態の確定

- メモリ上の`project.updatedAt`を保存確定時刻へ更新する。
- 基準`project.json`のSHA-256、`lastSavedFingerprint`、および履歴タブの保存位置を更新する。
- 編集`revision`とUndo／Redo履歴は変更しない。
- 手動保存成功を利用者へ確定する。

### ステージ6: 確定後処理

1. 現在状態とUndo／Redo履歴のどちらからも参照されない旧アセットを削除する。
2. 対応するRecovery JSONと不要なステージングファイルを削除する。
3. Catalogの最終確認名、作成日時、および更新日時キャッシュを更新する。
4. 自動バックアップの要否を判定し、必要な場合は別トランザクションとして開始する。

- 旧アセット、Recovery、一時ファイル、またはCatalogキャッシュの清掃・更新失敗で
  Project手動保存の成功を取り消さない。
- 不要ファイルを削除できない場合は安全な警告を表示し、次回起動または保存で再試行する。
- Catalog更新失敗時もProject本体を巻き戻さず、`.bak`からの復旧または再登録を可能にする。

## 手動保存失敗時の利用者操作

- 確定前の失敗では、旧正式データ、現在のメモリ編集状態、Undo／Redo履歴、および
  Recoveryを維持する。
- 利用者へ「再試行」「別の保存先へ保存」「キャンセル」を提示する。
- 「再試行」は保存先、ロック、外部変更、および空き容量を最初から再検査する。
- 「別の保存先へ保存」は基本設計に従い、新しいProject UUIDとProject日時を持つ
  独立Projectとして確定する。
- 「キャンセル」は編集を続行できる状態なら画面へ戻し、切断・ロック喪失状態では
  編集停止を維持する。

## Catalogトランザクション

1. 現在のCatalog候補を構築してSchemaとJava業務条件を検証する。
2. 現在の正常な`catalog.json`を検証して`catalog.json.bak`の一時名へコピーする。
3. `.bak`候補を同期・再検証してから、正式な`.bak`へ原子的置換する。
4. Catalog候補を同じディレクトリの一時JSONへ出力・同期・再検証する。
5. 一時JSONを`catalog.json`へ原子的置換する。

- 新しいCatalogの確定前に失敗した場合は旧Catalogを維持する。
- 起動時は`catalog.json`、次に`.bak`を検証し、両方利用不能なら空のCatalogを再作成できる
  画面へ移行する。Project本体を削除または変更しない。

## Recoveryトランザクション

- 未保存変更がある間、最後のRecovery確定から30秒以上経過した時点で最新状態を取得する。
- Recovery作成中もProjectコマンドの適用を長時間停止しないよう、同一`revision`の
  変更不能な保存モデルスナップショットを取得してからファイル処理を行う。
- Recovery固有Schema、埋め込んだProject Schema、Java整合性、およびステージングアセットを検証する。
- Recoveryディレクトリの一時JSONへ出力・同期・再検証し、正式な
  `<project-id>.recovery.json`へ原子的置換する。
- Recovery処理中に編集`revision`が進んでも、取得済みスナップショットを有効な
  直前状態として保存し、次の30秒周期で最新状態へ更新する。
- Recovery失敗だけを理由に正式Projectを変更せず、編集を継続できる場合は警告して
  次の周期で再試行する。

## Recoveryの適用

1. Recovery固有Schema、埋め込まれたProject Schema、Java業務条件、および
   必要なステージングアセットをすべて検証する。
2. 正式ProjectのUUIDとSHA-256がRecoveryの基準情報に一致することを確認する。
3. 保存済みProjectの更新日時、Recovery作成日時、および最大30秒程度前の状態である
   可能性を示して、復旧または破棄を利用者へ確認する。
4. 復旧を選択した場合は、スナップショットを新しい`ProjectEditingSession`の
   未保存状態として読み込む。
5. `revision`を0、Undo／Redoスタックを空、`lastSavedFingerprint`を正式Projectの
   フィンガープリントとして開始する。

- Recovery適用時点では正式`project.json`と正式アセットを上書きしない。
- 利用者は復旧内容を確認して手動保存するか、「変更を破棄」で正式保存状態へ戻れる。
- 復旧内容の手動保存成功後にRecovery JSONと不要なステージング領域を削除する。
- 復旧内容の変更破棄時もRecovery JSONと対応するステージング領域を削除する。
- Recovery、Projectスナップショット、または必要アセットの1件でも検証に失敗した場合は、
  部分適用せず正式保存状態を開く。
- 検証失敗したRecoveryを自動削除せず、個人情報を含まない理由で利用不能と通知する。
- 正式ProjectのUUIDまたはSHA-256が一致しない場合は、基本設計に従って
  別Projectとしての復旧またはRecovery破棄だけを許可する。

## 自動バックアップトランザクション

1. 手動保存成功後の正式Projectを対象として要否を判定する。
2. Projectロック、接続、空き容量、および正式データの整合性を確認する。
3. `backup/automatic`内の`.backup-<random-uuid>.tmp`へZIPをストリーミング作成する。
4. ZIP内のマニフェスト、Project JSON、アセット数、サイズ、およびSHA-256を検証する。
5. 一時ZIPを同期し、`backup-<UTC日時>.zip`へ原子的に変更する。
6. 正常な新世代の確定後だけ、4世代目以降を削除する。

- 正式`.zip`名への変更をバックアップの確定点とする。
- 失敗時は既存3世代を維持し、手動保存成功を取り消さない。
- 失敗を警告し、最終成功日を更新せず、次の手動保存成功後に再試行する。

## バックアップ復元トランザクション

### 事前検査と退避

1. 未保存変更がある場合は保存、破棄、またはキャンセルを求める。保存成功後または
   変更破棄で正式保存状態へ戻した後だけ復元を続行し、キャンセル時は開始しない。
2. 選択バックアップのZIP構造、マニフェスト、形式バージョン、Project Schema、
   業務不変条件、および全アセットのサイズ・SHA-256を一時領域で検証する。
3. Projectロック、接続、基準JSONのSHA-256、および必要空き容量を確認する。
4. 現在の正常な`project.json`と参照アセットを`backup/pre-restore`の一時ZIPへ格納する。
5. 復元前退避を再検証し、正式`.zip`名へ原子的に変更する。

- 選択バックアップまたは復元前退避の検証に失敗した場合は、正式データを変更せず終了する。
- 復元前退避の正常な新世代が確定するまで、既存3世代を削除しない。

### 復元アセットの配置

- 選択バックアップから、復元後のProjectが参照する全アセットをステージング領域へ展開する。
- パストラバーサル、リンク、未宣言ファイル、サイズ、MIMEタイプ、およびSHA-256を再検証する。
- 現在の正式領域に同じUUIDと同じSHA-256のアセットがある場合は再コピーせず再利用する。
- 同じUUIDでSHA-256が異なる場合は、不変IDの内容衝突として復元を中止する。
- 正式領域にない復元アセットは、手動保存と同じ一時コピー、同期、原子的移動、
  再読込検証で先に配置する。
- この段階で失敗した場合は現在の`project.json`を維持し、新規配置した未参照アセットを
  清掃対象とする。

### 復元Projectの確定

1. 選択バックアップ内の`project.json`をProjectルートと同じディレクトリの一時JSONへ出力する。
2. 一時JSONを同期し、Schema、Java業務条件、およびアセット参照を再検証する。
3. 確定直前にロック、接続、および現在の基準JSONのSHA-256を再確認する。
4. 一時JSONを正式`project.json`へ原子的置換する。
5. 置換後JSONを再読込し、期待したSHA-256と一致することを確認する。

- `project.json`の原子的置換を復元の確定点とする。
- アセットを不変UUIDの追加ファイルとして先に配置し、参照の正をJSONで切り替えることで、
  ディレクトリ全体の非原子的な入れ替えを行わない。
- 確定後は選択バックアップのProject UUID、名称、作成日時、および更新日時を使用する。
- Catalogの保存先とデフォルト設定は変更しない。
- 新しい`ProjectEditingSession`を`revision` 0、空のUndo／Redo履歴、
  復元Projectの保存済みフィンガープリントで開始する。
- 復元後のProjectと履歴から参照されない旧アセットは確定後の清掃対象とし、
  削除失敗で復元成功を取り消さない。

## 切断・ロック喪失時の確定状態

- 原子的置換を呼び出す前に切断またはロック喪失を検出した場合は`NOT_COMMITTED`とし、
  旧正式状態を正とする。
- 原子的置換が正常終了し、置換後JSONのSHA-256検証にも成功した場合は`COMMITTED`とする。
- 原子的置換の呼び出し中または直後に接続が失われ、結果を検証できない場合は
  `COMMIT_OUTCOME_UNKNOWN`として編集と追加書込みを停止する。
- `COMMIT_OUTCOME_UNKNOWN`では同じ保存処理を自動再試行しない。再接続後にロックを再取得し、
  ディスク上の正式JSONが旧SHA-256、新SHA-256、またはどちらでもないかを判定する。
- 新SHA-256なら確定済み、旧SHA-256なら未確定としてメモリ状態を整合させる。
- どちらでもない場合は外部変更または破損として元保存先へ上書きせず、別保存または
  変更破棄を求める。

## 起動時の残存ファイル処理

- 正式ファイル名ではない既知の一時命名規則だけを清掃候補とする。
- 一時ファイルが現在のプロセスで使用中でないこと、通常ファイルであること、
  シンボリックリンクでないこと、および管理領域内であることを確認する。
- 一時Project JSON、アセット、Catalog、Recovery、およびバックアップを正式データとして
  自動採用しない。
- `project.json`から参照されないUUID形式の正式アセットは、Undo／Redo履歴が存在しない
  起動時に再検証してから清掃対象とする。
- UUID形式でない不明ファイルは削除せず、読み込み対象外として安全な警告を表示する。

## 検証要件

- 各ステージの前後で強制終了、容量不足、権限拒否、ロック喪失、および切断を発生させ、
  確定点より前は旧状態、確定後は新状態だけが正式状態になることを確認する。
- 変更なしの保存でファイル、日時、リビジョン、履歴、およびバックアップが変わらないことを確認する。
- 新規アセット配置後かつJSON確定前の失敗で、旧Projectが正常に開けることを確認する。
- JSON確定後の旧アセット清掃失敗で保存成功となり、次回に未参照ファイルだけを清掃できることを確認する。
- 保存失敗後に編集状態とRecoveryを維持し、再試行、別保存、およびキャンセルを選べることを確認する。
- Recovery適用後に正式ファイルが変化せず、未保存状態、revision 0、空の履歴で開始することを確認する。
- Recovery適用後の手動保存と変更破棄で、対応するRecoveryと一時アセットが安全に清掃されることを確認する。
- Recoveryの一部が不正な場合に部分復旧せず、正式保存状態を変更しないことを確認する。
- Project保存で`project.updatedAt`だけが確定時刻へ変わり、編集`revision`と履歴が変わらないことを確認する。
- Catalog、Recovery、およびバックアップの一時ファイルが正式データとして誤認されないことを確認する。
- Backup失敗でProject保存成功と既存バックアップ3世代が維持されることを確認する。
- 復元アセット配置中の失敗で旧Projectが開け、JSON確定後にだけ復元状態へ切り替わることを確認する。
- 原子的置換中の切断で結果不明状態となり、自動再試行せず、再接続後の新旧SHA-256で
  確定状態を判定することを確認する。
- ローカル、外付け、および検証済みSMB 3.x共有で同じ障害注入試験を実行する。

# 検索設計

## 目的

1プロジェクト5,000墓所の編集状態に対して、利用者が入力した表記の揺れを吸収しながら、
該当する墓所を1秒以内に確認できる単一キーワード検索を提供する。

## 検索対象

- エリア名
- 管理番号
- 墓所名
- 墓所に紐づく人物の氏名および戒名
- 墓所に紐づく添付の表示名および説明

墓所の備考、添付ファイル本体、画像内文字、およびPDF変換画像内の文字はMVPの検索対象外とする。

## 正規化

保存値は変更せず、検索用文字列と検索キーワードにだけ同じ正規化を適用する。

1. Unicode NFKC正規化で全角・半角などの互換文字を統一する。
2. Unicodeの大文字・小文字を検索用に統一する。
3. 半角スペース、全角スペース、およびその他のUnicode空白文字を除去する。
4. カタカナを対応するひらがなへ変換する。

- 正規化後のキーワードが空の場合は検索条件なしとして扱う。
- 正規化後の検索対象文字列にキーワードが含まれる場合を部分一致とする。
- 元の表示値とJSON保存値は正規化結果で上書きしない。

## 検索処理

- Javaバックエンドの現在の`ProjectAggregate`を検索対象の正とする。
- 墓所ごとに検索対象文字列を関連データから構築し、正規化した検索用データをメモリ上で保持する。
- 人物全件を検索のためにReactへ常時送信せず、Java側で人物の検索文字列を墓所単位に集約する。
- 対象データの追加、変更、削除、Undo、およびRedoに合わせて、影響する墓所の検索用データを更新する。
- いずれかの対象項目が一致した場合に、その項目が属する墓所を結果へ追加する。
- 複数項目、複数人物、または複数添付が一致しても、墓所UUIDで重複を除去する。
- 検索、結果選択、および結果表示は保存対象外とし、編集リビジョンとUndo履歴を変更しない。

## 結果の並び順

検索結果は次の優先順で安定して並べる。

1. エリアの`displayOrder`
2. 管理番号の自然順
3. 墓所名
4. 墓所UUID

- 自然順では、数字の連続部分を数値として比較し、`1`、`2`、`10`の順にする。
- エリアの`displayOrder`は作成順を表し、新規エリアを末尾へ追加する。名称変更では変更せず、
  削除時は残りを詰め、Undo時は削除前の位置へ戻す。
- 英字の大小、全半角、およびかな表記は検索時と同じ正規化結果で比較する。
- 未割当の墓所は、すべてのエリア所属墓所の後に配置する。
- 未採番の墓所は、同一エリア内の採番済み墓所の後に配置する。
- 並び順は検索結果表示と最初のズーム対象を決めるためだけに使用し、保存データや地図座標を変更しない。
- 利用者が並び替え方法を切り替える機能はMVPに含めない。

## UI

- 単一のキーワード入力欄を表示する。
- 検索結果一覧は利用者が閉じるか、検索条件をクリアするまで保持する。
- 検索後は最初の結果へズームする。
- 結果を選択した場合は該当墓所を地図上で選択し、確認しやすい位置へ表示する。
- 数値・日付範囲、項目別条件、およびAND／OR指定はMVPに含めない。

## 検証要件

- 各対象項目の部分一致と、対象外項目が一致しないことを確認する。
- 英字大小、全半角、Unicode空白、およびひらがな・カタカナの違いを吸収できることを確認する。
- 同じ墓所の複数項目が一致しても、結果が1件になることを確認する。
- 数字を含む管理番号が自然順となり、未採番と未割当が所定の位置へ表示されることを確認する。
- 同じ条件の検索で安定した順序と最初のズーム対象が得られることを確認する。
- データ変更、Undo、およびRedo後に検索結果へ最新状態が反映されることを確認する。
- 5,000墓所の想定データで検索結果を1秒以内に返せることを確認する。
- 20,000人を含む5,000墓所の想定データでも検索結果を1秒以内に返せることを確認する。
- 検索操作で編集リビジョン、未保存状態、およびUndo履歴が変化しないことを確認する。

# データ保存・プライバシー設計

## 目的

Hakamapの構造化データ、画像、添付ファイル、バックアップをユーザー単位の
ローカル領域へ安全に保存し、個人用オフラインアプリとしての保護範囲を定める。

## 決定事項

### 保存先

- Windowsでは`%LOCALAPPDATA%\Hakamap`をアプリケーションデータのルートとする。
- インストール先ディレクトリへ変更可能なデータを書き込まない。
- 初期版ではデータルートをユーザーが任意の場所へ変更する機能を設けない。
- Windows以外の開発環境では、OSごとのユーザーデータディレクトリ規約に従う。
- テストでは一時ディレクトリを使用し、実際のユーザーデータを読み書きしない。

Windowsでの基本構成は次のとおりとする。

```text
%LOCALAPPDATA%\Hakamap\
├─ database\
├─ projects\
├─ backups\
├─ runtime\
├─ temp\
└─ logs\
```

- `database`にはH2データベースを保存する。
- `projects`にはプロジェクト単位の画像および添付ファイルを保存する。
- `backups`には世代管理するバックアップを保存する。
- `runtime`には二重起動の調整など、起動中だけ有効な情報を保存する。
- `temp`には確定前の一時ファイルを保存する。
- `logs`には個人情報を含まない運用ログだけを保存する。

### データベース

- 構造化データは、アプリケーション全体で1つの組み込みファイルモードH2
  データベースに保存する。
- H2のインメモリモードおよびサーバーモードを本番では使用しない。
- Hibernateの本番設定は`ddl-auto=validate`とする。
- スキーマの作成と変更は、バージョン管理されたFlywayマイグレーションだけで行う。
- 本番環境ではH2 Consoleを有効にしない。
- 保存処理は短いトランザクションとして実行し、確定済みの変更を速やかに
  データベースへ反映する。

### プロジェクトとアセット

- 各墓地プロジェクトへUUIDを割り当てる。
- 画像および添付ファイルはプロジェクト単位のディレクトリへ分離して保存する。
- データベースには端末固有の絶対パスを保存せず、管理対象領域内の相対参照を保存する。
- ファイルの書き込みは一時ファイルで完了させた後、確定先へ移動する。
- パストラバーサル、シンボリックリンク、その他の手段による管理対象領域外への
  読み書きを許可しない。
- プロジェクトの移動や共有には、将来提供する専用のエクスポートおよびインポート
  機能を使用する。
- バックアップ、復元、インポート、エクスポートでは、DBと参照先アセットを
  1つの整合性単位として扱う。

## プライバシーと脅威モデル

### 利用者向け認証

- Hakamapは単一ユーザー向けの個人用アプリとする。
- アプリ独自のアカウント、ログイン画面、ログインパスワードを設けない。
- DB、画像、添付ファイル、バックアップへアプリ独自の暗号化を適用しない。
- Windowsのユーザー認証とファイルアクセス制御を、保存データの主な保護境界とする。

### ファイルの保護

- アプリケーションデータ領域は、現在のWindowsユーザーだけが利用できるようにする。
- 作成するDB、アセット、バックアップ、一時ファイル、起動情報について、意図せず
  他の一般ユーザーへアクセスを許可しない。
- エクスポート先など、ユーザーが明示的に選択した外部の保存先については、その
  保存先のアクセス権が適用されることを画面上で説明する。

### HTTPセッションとの区別

- ローカルHTTP境界で使用する一時セッションは、利用者を識別するログイン機構ではない。
- 一時セッションは、外部Webサイトや無関係なローカルプロセスからAPIを操作されにくく
  するため、ランチャーとブラウザの間で自動的に確立する。
- 利用者に一時セッション用の認証情報の入力を要求しない。

### 保護対象外

次の主体によるデータアクセスは、初期版の保護対象外とする。

- 現在のWindowsユーザーと同じ権限で実行されるマルウェア
- 端末の管理者権限を持つ利用者またはプロセス
- OSの認証やディスクを迂回してストレージを直接解析できる主体
- ユーザー自身がアクセス制限のない場所へ出力したエクスポートまたはバックアップ

## ログ

- 氏名、日付、埋葬記録、メモ、ファイル内容、起動用認証情報をログへ記録しない。
- ファイルシステムの絶対パスを、通常の運用ログやAPIレスポンスへ含めない。
- 障害調査に必要な場合は、個人情報を含まないエラー分類、処理種別、内部IDなどを記録する。

## 検証要件

- Windowsでデータが`%LOCALAPPDATA%\Hakamap`配下へ作成されることを確認する。
- インストール先ディレクトリへ変更可能なデータが作成されないことを確認する。
- 本番設定でH2が組み込みファイルモードとなり、H2 Consoleが無効であることを確認する。
- 本番設定でFlywayの適用後にHibernateのスキーマ検証が成功することを確認する。
- テストが実際のユーザーデータディレクトリへアクセスしないことを確認する。
- APIへ絶対パスが露出しないことを確認する。
- 管理対象外を示す相対パス、絶対パス、シンボリックリンクを使用したファイル操作が
  拒否されることを確認する。

## 採用しない方式

- インストール先へのユーザーデータ保存
- H2のインメモリモードまたはサーバーモードによる本番運用
- Hibernateの`ddl-auto=update`による本番スキーマ変更
- 端末固有の絶対パスの永続化
- 初期版でのアプリ独自アカウント、パスワード、保存データの独自暗号化

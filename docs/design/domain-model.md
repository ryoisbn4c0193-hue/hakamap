# ドメインモデル・不変条件詳細設計

## 目的

Hakamapの業務状態をJavaバックエンドで一貫して管理するため、集約境界、エンティティ、
値オブジェクト、および変更時に必ず維持する不変条件を定める。

本書はメモリ上のドメインモデルを対象とする。JSON上の配置は
`data-storage-and-privacy.md`、具体的なJSON SchemaとJava保存モデルは後続の詳細設計で定める。

詳細設計上の内部技術判断は、確定済みの要件と基本設計を変更しない範囲で本書に定める。
利用者の操作、業務ルール、保存データ、または将来の互換性へ影響する判断は、
関連事項をまとめて利用者の確認を得る。

## 集約境界

### Project集約

- `Project`を、開いているプロジェクトの編集データ全体を管理する集約ルートとする。
- 背景、エリア、墓所、人物、およびアセットメタデータをProject集約の構成要素とする。
- 人物と添付アセットは、ドメイン上では墓所に所有される構成要素として扱う。
- 背景アセットは、ドメイン上では背景配置に所有される構成要素として扱う。
- JSONでは参照の重複を避けるため各データをトップレベル配列へ正規化して保存できる。
  ドメイン上の所有関係とJSON上の物理的な配置を同一とはみなさない。
- すべての編集コマンドはProject集約を通して実行し、構成要素をリポジトリから個別に
  更新する経路を設けない。
- Project集約は、コマンド適用後の状態が保存可能な不変条件を満たすことを保証する。
- 未採番、未割当、またはエリアからのはみ出しなど、要件上保存可能な状態は
  不変条件違反ではなく、導出した業務状態または警告として扱う。

### Catalog集約

- `Catalog`を、端末に登録されたプロジェクト一覧とデフォルトプロジェクトを管理する
  Project集約とは別の集約とする。
- CatalogはプロジェクトUUIDと端末上の保存先を関連付けるが、プロジェクトの業務データを
  複製して保持しない。
- Project集約の保存とCatalog集約の保存は、それぞれ独立した整合性単位とする。
- プロジェクト作成、インポート、再関連付け、削除、および復元など両集約にまたがる処理は、
  アプリケーションサービスが確定順序と失敗時の補償を制御する。

### Recovery

- 自動復旧データは独立した業務集約とせず、未保存のProject編集状態を救済する
  スナップショットとして扱う。
- 復旧データを正式状態へ反映する場合は、通常のProject読み込みと同じ構造検証および
  業務不変条件の検証を行う。

## 識別と参照

- Project、エリア、墓所、人物、およびアセットは、作成後に変更されないUUIDで識別する。
- API、コマンド、および集約内部の参照にはUUIDを使用する。
- エリア名、管理番号、墓所名、人物名、およびファイル表示名を参照識別子として使用しない。
- 添付と人物は、存在する墓所だけを所有元として参照できる。
- 背景配置は、背景種別のアセットだけを参照できる。
- UUIDの発行はJavaバックエンドで行い、フロントエンドが確定UUIDを指定する更新経路を設けない。

## アセット実体との境界

- Project集約が保持するのはアセットのUUID、種別、相対パス、表示情報、検証済みメタデータ
  などの業務状態とする。
- 背景および添付のバイナリ実体はProject集約へ読み込まず、アセットリポジトリを介して扱う。
- Project集約の参照整合性と、アセットリポジトリ上の実体の存在・ハッシュ整合性を
  保存トランザクションで一体として検証する。
- Project集約だけの変更では正式ファイルを直接操作せず、ステージング、保存確定、
  Undo／Redo、および変更破棄との順序をアプリケーションサービスが制御する。

## 責務の分離

- ドメインモデルは、業務上の作成・変更・削除、正規化、参照整合性、一意性、
  配置制約、および導出状態を担当する。
- アプリケーションサービスは、コマンド実行、権限のないローカルセッション検証、
  リビジョン照合、Undo／Redo登録、保存処理、および複数集約間の処理順序を担当する。
- リポジトリは、検証済み集約の読み書きと保存モデルへの変換を担当し、業務判断を行わない。
- コントローラーはDTO変換と入力形式の検証を担当し、業務ルールを重複実装しない。
- ReactとPixiJSは表示と入力を担当し、Java側の不変条件を正として扱う。

## Project集約に共通する不変条件

- 集約内の全エンティティUUIDは、種類をまたいで重複しない。
- 人物および添付の所有先墓所が存在する。
- 背景配置がある場合、参照する背景アセットが存在する。
- 保存座標、寸法、倍率、および角度は有限値であり、基本設計で定めた範囲と
  小数第3位への正規化を満たす。
- エリア数、エリア名、色、表示順、エリア同士の非重複を満たす。
- 墓所同士は面積を持って重ならず、辺または角の接触だけを許可する。
- 管理番号が設定され所属エリアがある墓所では、エリアUUIDと管理番号の業務キーが一意である。
- 同一墓所内の人物および添付の表示順は、0から始まる重複のない連番である。
- Project集約から導出できる所属エリア、完成状態、未割当状態、および警告状態を
  独立した永続プロパティとして保持しない。

## コマンド適用の原子性

- 単体操作と一括操作のどちらも、1つのコマンドを業務上の原子的な変更単位とする。
- 一括作成、移動、コピー、削除、および採番では部分成功を設けない。
- コマンドは変更対象をすべて反映した候補状態を構築し、その状態に対して不変条件と
  導出状態を検証する。
- 候補状態の構築中にだけ発生する重なりや順序の重複などの中間状態は外部へ公開しない。
- 候補状態に1件でも不変条件違反がある場合は、コマンド全体を反映しない。
- コマンド成功時は変更結果を一度に公開し、Undo／Redo履歴へ1操作として登録する。

## エラーと警告

- 墓所またはエリアの重なり、業務キー重複、不正な寸法、参照先欠落など、
  不変条件に違反する状態をエラーとして扱い、コマンド全体を拒否する。
- 未採番、未割当、および墓所全体が所属エリアに収まらない状態は警告として扱い、
  コマンドの反映と保存を許可する。
- 警告は候補状態全体から導出し、コマンド成功結果に含める。
- エラーがある場合も候補状態は公開せず、適用前のProject集約を維持する。

## 墓所の導出状態

- 墓所の状態は、完成状態と警告理由を独立した2軸で導出する。
- 完成状態は`COMPLETE`または`INCOMPLETE`とする。
- 所属エリアがない場合は未完成理由`UNASSIGNED`を付与する。
- 管理番号がない場合は未完成理由`UNNUMBERED`を付与する。
- 複数の未完成理由に該当する場合は、該当する理由をすべて返す。
- 所属エリアがあり、管理番号が設定され、業務キーの一意性を満たす場合は`COMPLETE`とする。
- 墓所の中心点が所属エリア内にあるが、墓所全体がエリア内に収まらない場合は、
  完成状態を変更せず警告理由`OUTSIDE_AREA_BOUNDS`を付与する。
- エリアの`visible`は表示だけに影響し、所属判定、業務キー、および完成状態には使用しない。
- エリアを削除した場合は残存エリアから所属を再計算し、候補がなければ`UNASSIGNED`とする。
- エリア削除のUndo／Redoでも、復元された候補状態全体から所属、完成状態、および警告を再計算する。
- 完成状態、未完成理由、および警告理由は導出値とし、`project.json`へ保存しない。

## 墓所コピー

- 墓所コピーは、既存墓所の区画形状を使用して情報未入力の新しい墓所を作成する操作とする。
- コピー先には幅、高さ、および内部回転角度を引き継ぎ、位置はコピー操作で指定した
  移動量から算出する。
- 管理番号、墓所名、備考、人物、および添付はコピーしない。
- コピー先の墓所UUIDはJavaバックエンドで新規発行し、コピー元との関連を保存しない。
- コピー先の`updatedAt`にはコピーコマンドのUTC日時を設定する。
- 複数墓所の一括コピーでは、コピー元同士の位置関係を維持し、すべてに同じ移動量を適用する。
- コピー後の墓所同士、既存墓所、およびエリアとの関係は、全コピー先を含む候補状態で検証する。
- 1件でも不変条件違反がある場合はコピー全体を拒否し、コピー先を1件も作成しない。
- コピー元の墓所、人物、添付、および更新日時は変更しない。

## 墓所の作成と一括生成

### 単体作成

- 地図上のドラッグ開始点と終了点から、墓所の左上座標、幅、および高さを算出する。
- 座標と寸法を保存精度へ正規化し、幅または高さが0以下になる場合は作成を拒否する。
- 作成候補を含むProject全体で重なりなどの不変条件を検証し、成功時だけ墓所を追加する。

### 行列指定による一括作成

- 入力は、行数、列数、1墓所の幅と高さ、横方向と縦方向の間隔、および左上の開始位置とする。
- 左上の開始位置から、左から右、上から下へ行優先で墓所を生成する。
- 横方向の配置間隔は墓所の幅と横間隔の合計、縦方向の配置間隔は墓所の高さと
  縦間隔の合計から算出する。

### 指定範囲の埋め生成

- 入力は、地図上で指定した長方形範囲、1墓所の幅と高さ、および横方向と縦方向の間隔とする。
- 指定範囲の左上から、左から右、上から下へ行優先で墓所を生成する。
- 正規化後の墓所長方形全体が指定範囲内へ収まる候補だけを作成する。
- 指定範囲の右端または下端で一部だけが収まる墓所は、切り詰めず作成しない。
- 正規化後に1件も配置できない場合は、作成対象がないことを示してコマンドを反映しない。

### 共通の生成規則

- 生成順は左から右、上から下とし、座標が同一の場合はUUIDで安定した順序を得る。
- 生成順は一括採番の初期順序として使用するが、永続的な墓所表示順として保存しない。
- 生成する各墓所には新しいUUID、回転角度0、および同じコマンドUTC日時の`updatedAt`を設定する。
- 管理番号、墓所名、備考、人物、および添付は設定しない。
- 生成候補のいずれかに不正な値または重なりがある場合は、生成全体を拒否する。
- 未割当および所属エリアからのはみ出しは警告として返し、生成を許可する。

## 一括採番

- 一括採番の入力は、任意の接頭辞、0以上の開始番号、1～10の数字桁数、および
  任意の接尾辞とする。MVPでは増分を1に固定する。
- 数字部分は指定桁数まで先頭を0で埋める。指定桁数を超える数値は切り捨てず、
  実際の桁数で表現したうえで管理番号全体の文字数上限を検証する。
- 管理番号が設定済みの墓所が選択対象に1件でも含まれる場合は、既存番号を上書きせず
  コマンド全体を拒否する。
- 未割当墓所が選択対象に1件でも含まれる場合は、業務キーを確定できないため
  コマンド全体を拒否する。
- 選択墓所は視覚的な行へまとめ、行内を左から右、行を上から下へ並べる。
- 採番を確定する前に、墓所UUID、画面表示用の墓所識別情報、および割当予定番号を
  順序付きでプレビューする。
- MVPではプレビュー上の手動並べ替えを設けない。
- 複数エリアにまたがる選択を許可し、番号列はエリア境界でリセットせず継続する。
- 一意性は各墓所の所属エリアUUIDと正規化した生成番号の組み合わせで検証するため、
  異なるエリアでは同じ番号を許可する。
- 生成した管理番号の文字数超過、既存業務キーとの重複、または選択墓所間の
  業務キー重複が1件でもある場合は、全墓所を変更しない。
- 採番成功時は対象墓所へ同じコマンドUTC日時を設定し、1件のUndo／Redo操作として扱う。
- 視覚的な行の厳密な判定アルゴリズムは、地図上のプレビューとバックエンドの採番結果が
  一致するよう、PixiJS・当たり判定の詳細設計で定める。

## 更新日時

- 1つのコマンドで変更する全エンティティには、Javaバックエンドがコマンド開始時に発行した
  同じUTC日時を使用する。
- 人物または添付の追加、変更、削除、および並べ替えでは、所属墓所の`updatedAt`も
  同じコマンド日時へ更新する。
- 複数墓所に影響する一括操作では、対象墓所へ同じコマンド日時を設定する。
- `project.updatedAt`は編集コマンドでは変更せず、データ変更を含む手動保存の成功時に更新する。
- Undoは、更新日時を含めてコマンド適用前の状態へ戻す。
- Redoは最初のコマンド適用時に設定した更新日時を再適用し、Redo実行時刻で置き換えない。

## 文字列の正規化と一意性

- 表示用文字列は、項目ごとの要件に従って前後の半角・全角空白を除去し、途中の空白、
  全角・半角、英字の大小、およびひらがな・カタカナは入力表記のまま保存する。
- 任意項目は前後空白の除去後に空文字となる場合、空文字を保持せず未設定として扱う。
- エリア名と管理番号の一意性判定では、保存表示値から次の比較キーを導出する。
  1. Unicode NFKCで互換文字を正規化する。
  2. Unicodeの大文字・小文字を比較用に統一する。
  3. 半角・全角を含むすべてのUnicode空白文字を除去する。
  4. ひらがなとカタカナを比較用に統一する。
- エリア名は、上記の比較キーがプロジェクト内で重複しないことを要求する。
- 管理番号は、所属エリアUUIDと上記の比較キーの組み合わせが重複しないことを要求する。
- 比較キーは導出値とし、Project集約および`project.json`へ永続化しない。
- 検索と一意性判定は同じ正規化サービスを使用し、正規化規則を重複実装しない。

## 数値の正規化と幾何判定

- 座標、寸法、倍率、および角度は、コマンドの候補状態を構築するときに基本設計で定めた
  保存精度へ正規化する。
- 重なり、接触、所属エリア、および範囲外警告は、正規化後の候補値で判定する。
- 正規化によって幅または高さが0以下になる場合は不正な寸法としてコマンド全体を拒否する。
- 正規化後に墓所またはエリアが面積を持って重なる場合は、画面上の正規化前候補で
  重なっていなくてもコマンド全体を拒否する。
- APIの成功結果には正規化後の確定値を含め、フロントエンドは入力前の候補値で上書きしない。

## Java内部の保持方針

- Project集約内のエリア、墓所、人物、およびアセットは、UUIDをキーとするMapで保持する。
- Mapの反復順序に業務上の意味を持たせず、必要な表示順は`displayOrder`などの明示値から導出する。
- 集約外へ変更可能なMapまたはListを公開せず、読み取り専用のビューまたは不変コピーを返す。
- エンティティのUUIDは変更不能とし、業務目的を表さない汎用setterを設けない。
- 一括コマンドは変更後のMapを候補状態として構築し、検証成功後にだけ集約の正式状態とする。

## Javaドメイン型

### 集約とエンティティ

| Java型 | 種別 | 主な責務 |
| --- | --- | --- |
| `ProjectAggregate` | 集約ルート | 編集状態、構成要素、コマンド適用、不変条件の維持 |
| `ProjectMetadata` | エンティティ | プロジェクトUUID、名称、作成日時、保存済み更新日時 |
| `BackgroundPlacement` | エンティティ | 背景アセット参照、位置、回転、X・Y倍率 |
| `Area` | エンティティ | 名称、長方形、色、表示状態、表示順 |
| `Grave` | エンティティ | 管理番号、墓所名、備考、長方形、回転、更新日時 |
| `Person` | エンティティ | 所有墓所、氏名、戒名、日時、表示順 |
| `AssetMetadata` | エンティティ | 種別、所有先、相対パス、表示情報、検証済み属性、表示順 |
| `CatalogAggregate` | 集約ルート | 登録プロジェクト、保存先、デフォルト設定 |

- 各エンティティはUUIDをコンストラクターで受け取り、作成後に変更しない。
- エンティティの変更は、目的を表すメソッドまたはProject集約のコマンド適用処理からだけ行う。
- JPAアノテーション、引数なし公開コンストラクター、および永続化都合のsetterを設けない。
- 保存モデルとの相互変換はリポジトリアダプター内のMapperが担当し、ドメイン型へ
  JSONライブラリ固有のアノテーションを付けない。

### 識別子

- `ProjectId`、`AreaId`、`GraveId`、`PersonId`、および`AssetId`を、UUIDを包む
  変更不能なrecordとして定義する。
- 異なる種類のUUIDを同じメソッドへ誤って渡せないよう、裸の`UUID`をドメインAPIで使用しない。
- UUIDの文字列表現との変換はAPIおよび保存モデルの境界で行う。

### 文字列値オブジェクト

- `ProjectName`、`AreaName`、`ManagementNumber`、`GraveName`、`GraveNotes`、
  `PersonName`、`PosthumousName`、`AssetDisplayName`、および`AssetDescription`を
  項目別の値オブジェクトとする。
- 各値オブジェクトが、前後空白、空文字、制御文字、改行可否、およびUnicodeコードポイント
  単位の文字数上限を検証する。
- 任意項目は`Optional`で未設定を表し、空文字を有効値として保持しない。
- 一意性と検索に使う比較キーは`TextComparisonKey`として正規化サービスから生成する。

### 数値と幾何

- バックエンドの保存座標、寸法、倍率、および角度には`BigDecimal`を使用し、
  小数第3位へ正規化してからドメイン状態へ格納する。
- `MapPoint`、`MapSize`、`MapRectangle`、`RotationDegrees`、および
  `BackgroundScale`を変更不能な値オブジェクトとする。
- `MapRectangle`は左上座標と正の幅・高さを保持し、右端、下端、中心点、
  面積を持つ交差、接触、および包含判定を提供する。
- フロントエンドのJavaScript `number`との変換誤差はAPI境界で正規化し、
  成功レスポンスではバックエンドの確定値を返す。

### 日時、順序、列挙

- 日時は`Instant`で保持し、UTCのISO 8601文字列との変換を境界で行う。
- 表示順は0以上の`DisplayOrder`値オブジェクトとし、所有範囲内の連番検証は
  Project集約が行う。
- エリア色は`AreaColorPreset`、アセット種別は`AssetType`、完成状態は
  `GraveCompletionStatus`、未完成理由は`IncompleteReason`、警告理由は
  `DomainWarningCode`のenumとする。

## ドメインサービス

| Java型 | 責務 |
| --- | --- |
| `TextNormalizationService` | 検索・一意性に共通する比較キーの生成 |
| `AreaMembershipResolver` | 墓所中心点から所属エリアを一意に導出 |
| `GeometryConstraintService` | 墓所・エリアの重なり、接触、包含の検証 |
| `BusinessKeyService` | エリアUUIDと管理番号比較キーによる一意性検証 |
| `GraveStatusDeriver` | 完成状態、未完成理由、警告理由の導出 |
| `GraveGenerationService` | 行列作成、範囲埋め、コピー候補の生成 |
| `NumberingService` | 一括採番候補とプレビューの生成 |
| `ProjectInvariantValidator` | Project候補状態全体の参照・順序・上限・業務整合性検証 |

- 状態を必要としない処理をドメインサービスとし、サービス自身へ可変状態を保持しない。
- 個別サービスの検証を通過しても、コマンド確定前に`ProjectInvariantValidator`で
  候補状態全体を検証する。
- 同じ規則をエンティティ、サービス、コントローラーへ重複実装しない。

## ドメイン処理結果

- コマンドの内部結果は、成功または失敗を表すsealed interfaceとして定義する。
- 成功結果は、確定した候補状態、正規化後の変更値、導出警告、およびUndo／Redo差分の
  作成に必要な変更集合を含む。
- 失敗結果は、安定したドメインエラーコードと、対象を特定する個人情報を含まない
  エンティティUUIDだけを含む。
- 日本語の画面文言をドメイン結果へ格納せず、APIエラー変換とフロントエンドの
  メッセージ定義でコードから解決する。
- 複数の入力違反を安全に返せる場合はまとめて返すが、候補状態および入力された
  個人情報をエラー詳細へ含めない。
- 警告は成功結果に含め、エラーと同じ失敗型へ混在させない。

## パッケージ境界

```text
jp.hakamap.project.domain
├─ model
├─ value
├─ service
├─ command
└─ result

jp.hakamap.catalog.domain
├─ model
├─ value
└─ result
```

- ドメインパッケージはSpring MVC、Jackson、ファイルシステム、およびPixiJSの型へ依存しない。
- アプリケーションサービス、保存アダプター、およびWebアダプターからドメインへ依存し、
  ドメインから外側の層へ依存しない。

## 未決事項

- 個別のドメインエラーコード一覧は、API・DTO・エラーコードの詳細設計で確定する。

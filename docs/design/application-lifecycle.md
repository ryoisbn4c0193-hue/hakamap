# アプリケーションの起動・終了設計

## 目的

HakamapをWindows向けデスクトップアプリとして実行する際の、アプリケーションの
起動、再起動、ブラウザとの関係、および安全な終了に関する要件を定める。

## 決定事項

### 起動

- Windows向けランチャーからSpring Bootを起動する。
- Spring Bootの準備完了後に、ユーザーの既定ブラウザでHakamapを開く。
- ブラウザにはローカルHTTPサーバーが配信するReact画面を表示する。
- 2回目以降のランチャー起動ではバックエンドを重複起動せず、既に起動している
  Hakamapをブラウザで開く。
- 初期版ではタスクトレイ常駐機能を設けない。
- プロジェクトを開く場合は、データを読み込む前にJavaの`FileChannel.tryLock()`で
  `.hakamap.lock`の排他ロックを取得する。
  取得できない場合はプロジェクトを開かず、別のプロセスまたは端末で利用中の可能性を通知する。

### ブラウザとバックエンドの寿命

- ブラウザのタブを閉じても、バックエンドを自動終了しない。
- バックエンドの寿命を、ブラウザのタブ、ウィンドウ、セッションの状態に依存させない。
- 初期版では、ブラウザからの接続がないことを理由とするタイムアウト終了を行わない。
- アプリケーション画面に、Hakamap全体を明示的に終了する操作を設ける。

### 通常終了

- 明示的な終了操作では、新しい更新の受付を停止してからSpringのコンテキストと
  使用中の永続化リソースを正常に閉じ、バックエンドプロセスを終了する。
- 開いているプロジェクトの排他ロックは終了処理で解放し、`FileChannel`を閉じてから
  `.hakamap.lock`を削除する。
- Windowsから通常の終了通知を受け取った場合も、可能な範囲で同じ正常終了処理を行う。
- 明示的な終了操作のAPIは、外部サイトや別プロセスから容易に実行できないよう保護する。
  具体的な保護方式はローカルHTTP境界の設計で決定する。

### 開いているプロジェクトの削除

- 未保存変更がある場合は、保存してごみ箱へ移動、変更を破棄してごみ箱へ移動、または
  キャンセルを利用者に選択させる。
- 移動前にプロジェクトを閉じて排他ロックを解放する。
- 移動成功後はプロジェクト選択画面を表示し、閉じたプロジェクトのUndo／Redo履歴を破棄する。
- 移動失敗時は、可能であれば元の場所のロックを再取得してプロジェクトを開き直す。

### 異常終了とデータ保護

- OSの強制終了、電源断、プロセスの強制終了などでは、終了処理が実行されることを
  前提にしない。
- データの安全性を終了時の処理だけに依存させず、保存処理のトランザクションと
  起動時の復旧によって確保する。
- プロジェクトの通常保存は利用者が明示的に実行する。
- 未保存変更がある間は30秒ごとに、プロジェクト単位で最新1世代の復旧JSONを
  自動復旧用領域へ安全に置換して保存する。変更がなければ更新しない。
- 新規アセットはアプリケーション管理領域の編集セッション別一時領域へステージングし、
  手動保存の完了時にプロジェクトの`assets`へ反映する。
- 削除対象の正式アセットは手動保存が完了するまで保持し、Undoと変更破棄に利用する。
- JSONとアセットの更新が中断された場合に、不整合を検出できるようにする。
- 起動時に前回の異常終了と、JSON、正式アセット、およびステージング済みアセットの
  不整合を検出できるようにする。
- 編集中データの自動復旧情報を通常保存およびバックアップとは別に管理し、異常終了後の
  次回起動時に復元または破棄を利用者へ確認する。
- 手動保存または変更破棄の成功後は、対応する復旧JSONと不要な一時アセットを削除する。
- 復旧データの検証に失敗した場合は適用せず、正式保存データを変更しない。
- 復旧JSONには対象プロジェクトUUID、形式とアプリのバージョン、作成日時、基準となる
  正式JSONのSHA-256、および一時アセットのIDとSHA-256を保持する。
- 正式JSONのUUIDとSHA-256が一致する場合だけ元プロジェクトへ適用し、不一致時は
  別プロジェクトとしての復旧または復旧データの破棄を選択させる。
- 自動復旧情報にはステージング済みアセットとの関連を保持し、復旧または破棄の選択に
  合わせて再関連付けまたは削除する。
- 復旧対象でない不要なステージング領域は次回起動時に清掃する。
- 手動保存は、新規アセットのコピーと検証、一時JSONの出力と検証、`project.json`の置換、
  JSONとUndo／Redo履歴のどちらからも未参照となった旧アセットの削除、の順で行う。
  JSONの置換を保存の確定点とする。
- 保存確定前に中断した場合は旧JSONと旧アセットを維持し、確定前にコピー済みだった
  未参照アセットは次回起動時に清掃する。
- 自動清掃はJSONから参照されないUUID形式のアセットに限定する。UUID形式でない不明な
  ファイルは削除せず、無視して利用者へ警告する。
- 保存先の切断または排他ロック喪失時は、新しい編集操作と元保存先への保存を停止し、
  編集状態をメモリとローカル復旧領域に保持する。
- 再接続時は排他ロックを再取得し、基準として保持した`project.json`のSHA-256と
  ディスク上の値が一致する場合だけ編集を再開する。
- 外部変更がある場合は上書きせず、別の保存先への保存または変更破棄を求める。
- 別保存は新しいプロジェクトUUIDと日時で作成し、成功後は新しいプロジェクトへ切り替える。
  元プロジェクトとデフォルトプロジェクト設定は変更しない。
- 保存先と排他ロックは5秒ごとに`FileLock.isValid()`、ロックファイルの生存日時更新、
  および`FileChannel.force(true)`で監視し、通常監視の連続2回失敗で切断と判定する。
- データ変更、手動保存、バックアップ、およびエクスポートの直前にも確認し、失敗した
  操作は開始せず、接続復旧後も利用者の明示操作があるまで編集を再開しない。
- 確定済みデータの復旧手段として、世代管理されたバックアップを提供する。

## 今後決定する事項

- 単一起動を判定し、既存プロセスのURLを安全に取得する方式
- 異常終了を記録・検出する方式

## 採用しない方式

- ブラウザのタブを閉じたことだけを理由にバックエンドを終了する方式
- ブラウザとの通信切断だけを理由にバックエンドを終了する方式
- 初期版でのタスクトレイ常駐
- 正常終了処理が必ず実行されることを前提としたデータ保護

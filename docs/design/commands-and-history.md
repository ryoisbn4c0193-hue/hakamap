# コマンド・Undo／Redo差分詳細設計

## 目的

利用者の確定操作をJavaバックエンドの原子的なコマンドとして適用し、Project集約の
不変条件を維持したまま、最大100操作のUndo／Redoと未保存判定を提供する。

## 編集セッション

- 開いているProjectごとに`ProjectEditingSession`を1つだけ作成する。
- セッションは現在の`ProjectAggregate`、単調増加する`revision`、Undoスタック、
  Redoスタック、最後に保存した状態フィンガープリント、およびアセット保持情報を管理する。
- `revision`はセッション開始時に0とし、成功したデータ変更、Undo、およびRedoごとに1増やす。
- `revision`は同一性や日時ではなく、古い画面からの更新を拒否するセッション内の連番とする。
- `revision`、履歴、および状態フィンガープリントは`project.json`へ保存しない。

## コマンド契約

```java
interface ProjectCommand<R> {
  CommandType type();

  CommandEvaluation<R> evaluate(ProjectAggregate current, CommandContext context);
}

record CommandContext(
    long expectedRevision,
    Instant commandTimestamp,
    UuidGenerator uuidGenerator) {}
```

- コマンドは現在の集約を直接変更せず、候補状態、戻り値、および`ProjectChangeSet`を評価する。
- UUIDとコマンド日時はJavaバックエンドが`CommandContext`から供給する。
- 評価中に発行したUUIDと日時は成功した差分へ固定し、Redo時に再発行しない。
- `expectedRevision`が現在値と一致しない場合は評価を開始せず、競合として拒否する。
- 候補状態の全不変条件を検証し、成功した場合だけセッションの現在状態を一度に切り替える。
- 失敗、警告確認のキャンセル、および実質的な変更がない操作では、状態、リビジョン、
  更新日時、および履歴を変更しない。

## 配置警告の確認

- コマンドによって対象墓所へ`UNASSIGNED`または`OUTSIDE_AREA_BOUNDS`が新しく追加される場合は、
  候補状態を適用せず`ConfirmationRequired`結果を返す。
- `UNNUMBERED`だけが新しく追加される場合は確認を要求せず、成功結果の導出状態として返す。
- 操作前から存在する同じ配置警告が継続するだけの場合は、確認を繰り返さない。
- 複数墓所では警告コードごとの対象件数をまとめ、対象の管理番号、名称、人物情報などを
  確認文およびトークンへ含めない。
- 利用者が確認した場合は、サーバーが発行した単一使用の`confirmationToken`と
  元の`expectedRevision`を使用して確定を要求する。
- 警告候補は編集セッションのメモリへ最大5分保持し、トークン文字列自体には候補状態、
  UUID、入力値、および個人情報を埋め込まない。
- 確定時にリビジョン、トークンの有効期限、単一使用、および候補状態の不変条件を再確認する。
- リビジョンが変化した、期限切れ、使用済み、またはセッションが異なる場合は候補を適用せず、
  最新状態から操作をやり直すよう返す。
- キャンセル時は候補とトークンを破棄し、履歴とリビジョンを変更しない。
- Undo／Redoは確認済みの前後状態を復元するため、配置警告の確認を要求しない。
- 手動保存でも配置警告を再確認しない。

## MVPコマンド一覧

| 分類 | CommandType | 履歴の単位 |
| --- | --- | --- |
| プロジェクト | `RENAME_PROJECT` | 名称変更の確定1回 |
| 背景 | `SET_BACKGROUND` | 追加または差し替え1回 |
| 背景 | `TRANSFORM_BACKGROUND` | 1回のドラッグまたは数値適用 |
| 背景 | `REMOVE_BACKGROUND` | 確認後の削除1回 |
| エリア | `CREATE_AREA` | 作成1件 |
| エリア | `UPDATE_AREA` | 名称、形状、色、表示状態の確定1回 |
| エリア | `DELETE_AREA` | エリア1件の削除と導出状態再計算 |
| 墓所 | `CREATE_GRAVE` | ドラッグ作成1件 |
| 墓所 | `CREATE_GRAVE_GRID` | 行列指定の全生成墓所 |
| 墓所 | `FILL_GRAVE_RANGE` | 指定範囲の全生成墓所 |
| 墓所 | `UPDATE_GRAVE_INFO` | プロパティ適用1回 |
| 墓所 | `MOVE_GRAVES` | 単体または複数の1回の移動 |
| 墓所 | `RESIZE_GRAVE` | 1回のドラッグまたは数値適用 |
| 墓所 | `COPY_GRAVES` | 単体または複数の全コピー先 |
| 墓所 | `DELETE_GRAVES` | 確認した単体または複数の一括削除 |
| 墓所 | `NUMBER_GRAVES` | プレビュー確認後の全採番対象 |
| 人物 | `CREATE_PERSON` | ダイアログ保存1回 |
| 人物 | `UPDATE_PERSON` | ダイアログ保存1回 |
| 人物 | `DELETE_PERSON` | 確認後の削除1回 |
| 添付 | `ADD_ATTACHMENTS` | 1回の選択で追加した全ファイル |
| 添付 | `UPDATE_ATTACHMENT` | 表示情報の適用1回 |
| 添付 | `REORDER_ATTACHMENTS` | 1回の並べ替え確定 |
| 添付 | `DELETE_ATTACHMENT` | 確認後の削除1回 |

- Undo、Redo、手動保存、検索、選択、ズーム、パン、印刷、エクスポート、および
  バックアップは新しいChangeSetを作るCommandTypeに含めない。
- エリア更新で名称と形状など複数項目を同じフォームから同時確定した場合は
  `UPDATE_AREA`の1履歴とする。
- 背景差し替えと配置初期化を同時に行う場合は`SET_BACKGROUND`の1履歴とする。
- 背景差し替えでは現在の位置、回転角度、およびX・Y倍率を初期プレビューへ引き継ぎ、
  利用者が確定した配置を`SET_BACKGROUND`の変更後差分とする。
- 画像寸法が異なる場合も位置と倍率を自動補正せず、エリアと墓所の座標を変更しない。
- 背景削除の差分は背景配置と背景アセットメタデータを保持し、Undoで両方を復元する。
- `ADD_ATTACHMENTS`は選択した全ファイルを1候補として検証し、1件でもエラーがあれば
  添付メタデータとステージング状態を1件もProjectへ追加しない。

## 履歴へ記録する操作単位

### ポインター操作

- 墓所、エリア、および背景のドラッグ中はReactとPixiJSで候補をプレビューする。
- ポインターを離して操作を確定した時点で、開始状態から確定状態までを1コマンドとして送信する。
- ドラッグ中の各座標をAPIへ送信せず、履歴へも記録しない。
- ドラッグをキャンセルした場合、または確定値が開始値と同じ場合はコマンドを記録しない。

### フォームとダイアログ

- 入力中の各キー操作をコマンドにせず、「保存」または「適用」で1コマンドとして送信する。
- 1つの確定操作で複数項目を変更した場合も1履歴とする。
- キャンセルした入力、クライアント側検証で拒否した入力、およびサーバーで失敗した入力は
  履歴へ記録しない。

### 矢印キー移動

- キーを押してから離すまでの連続移動量をフロントエンドで集約し、キーを離した時点で
  1つの移動コマンドとして送信する。
- キーを離して再度押した場合は別コマンドとする。
- Shift併用と複数選択でも同じ境界を使用する。
- フォーカス喪失などで`keyup`を受け取れない場合は、その時点までの候補を確定せず
  操作開始位置へ表示を戻す。

### 一括操作

- 一括作成、範囲埋め、移動、コピー、削除、および採番は対象件数によらず1コマンドとする。
- 部分成功は設けず、成功時は全対象の前後差分を同じ履歴項目へ格納する。

## 差分モデル

```java
record ProjectChangeSet(
    CommandId commandId,
    CommandType commandType,
    Instant commandTimestamp,
    List<EntityDelta<AreaId, Area>> areaDeltas,
    List<EntityDelta<GraveId, Grave>> graveDeltas,
    List<EntityDelta<PersonId, Person>> personDeltas,
    List<EntityDelta<AssetId, AssetMetadata>> assetDeltas,
    ValueDelta<ProjectName> projectNameDelta,
    ValueDelta<BackgroundPlacement> backgroundDelta,
    Set<AssetId> retainedAssetIds) {}

record EntityDelta<I, T>(I id, T before, T after) {}
record ValueDelta<T>(T before, T after) {}
```

- 作成は`before`なし・`after`あり、削除は`before`あり・`after`なし、変更は両方ありで表す。
- 差分の`before`と`after`は変更不能なドメインスナップショットとする。
- 変更していないエンティティを差分へ含めず、Project全体のJSONコピーを保持しない。
- 墓所削除は、墓所、関連人物、および添付メタデータの各差分を1つのChangeSetへ含める。
- プロジェクト名称変更の差分には名称だけを含め、手動保存が管理する
  `project.updatedAt`をUndo／Redoで巻き戻さない。
- エリア変更による所属・完成状態・警告は導出値のため差分へ保存せず、適用後の状態から再計算する。
- 表示順の詰め直しで値が変わる全エンティティは差分へ含める。

## Undo

1. リクエストの`expectedRevision`を確認する。
2. Undoスタック最上位のChangeSetを取得する。
3. 各差分の`before`から候補Projectを構築する。
4. 参照、アセット保持、およびProject全体の不変条件を検証する。
5. 成功時だけ候補を現在状態とし、ChangeSetをRedoスタックへ移す。
6. `revision`を1増やし、導出状態と未保存判定を更新する。

- Undo操作自体を新しいChangeSetとして作成しない。
- 更新日時、UUID、および表示順を含め、操作直前の状態を正確に復元する。
- 復元に必要なアセット実体が欠落または破損している場合はUndo全体を拒否し、
  現在状態と履歴位置を変更せずデータ整合性エラーを返す。

## Redo

1. リクエストの`expectedRevision`を確認する。
2. Redoスタック最上位のChangeSetを取得する。
3. 各差分の`after`から候補Projectを構築する。
4. 参照、アセット保持、およびProject全体の不変条件を検証する。
5. 成功時だけ候補を現在状態とし、ChangeSetをUndoスタックへ戻す。
6. `revision`を1増やし、導出状態と未保存判定を更新する。

- 元の業務コマンドを再実行せず、保存した`after`を再適用する。
- UUID、コマンド日時、および採番結果をRedo時に再生成しない。
- 再適用に必要なアセット実体が欠落または破損している場合はRedo全体を拒否し、
  現在状態と履歴位置を変更しない。

## 新規操作と履歴上限

- Undo後に新しいコマンドが成功した場合は、既存のRedoスタックをすべて破棄する。
- Undoスタックへ101件目を追加する場合は、最も古いChangeSetを破棄する。
- ChangeSetを破棄する前に、そのChangeSetだけが保持しているアセット実体を算出する。
- 現在状態、残るUndo／Redo履歴、および保存済み状態のいずれからも参照されない
  アセットだけを清掃可能とする。
- プロジェクト終了時は両スタックを破棄し、正式保存状態から参照されない一時アセットを清掃する。

## 未保存判定

- 保存対象データを決定的な保存モデルへ変換し、SHA-256状態フィンガープリントを算出する。
- アセットのバイナリ全体を毎回読み直さず、検証済みアセットメタデータのUUID、相対パス、
  サイズ、およびSHA-256を保存モデルに含めてフィンガープリントへ反映する。
- 手動保存成功時に、実際に確定した`project.json`の保存モデルに対応する
  フィンガープリントを`lastSavedFingerprint`として更新する。
- 現在状態のフィンガープリントと`lastSavedFingerprint`が異なる場合だけ未保存変更ありとする。
- Undo／Redoまたは新規操作によって保存済みデータと同じ状態になった場合は、
  履歴位置にかかわらず未保存変更なしとする。
- ズーム、パン、選択、検索、履歴、リビジョン、および導出状態をフィンガープリントへ含めない。

## 保存位置の表示

- 手動保存はChangeSetを作成しない。
- 履歴タブの「保存済み」表示は、保存成功時点で適用中だったChangeSetの`commandId`を
  セッション情報として記録する。
- 履歴上限または分岐で該当ChangeSetが破棄された場合は、特定行の目印を表示せず、
  現在が保存済み状態と同じか異なるかだけを表示する。
- 保存位置の目印と未保存判定が食い違う場合は、状態フィンガープリントの判定を正とする。

## 履歴表示

- ChangeSetには個人情報を含まない`CommandType`と対象件数を保持する。
- 画面文言は「墓所3件を移動」「人物を1件削除」のようにCommandTypeと件数から生成する。
- 管理番号、墓所名、氏名、戒名、備考、ファイル名、および入力値を履歴説明へ含めない。
- 適用中のUndoスタックとUndo済みのRedoスタックを合わせ、操作日時の新しい順に最大100件表示する。
- 任意行へのジャンプ、履歴の個別削除、および手動全消去は提供しない。

## 検証要件

- ドラッグ、フォーム確定、キー押下区間、および一括操作が所定の履歴単位になることを確認する。
- 実質的に変更がない操作でリビジョン、日時、および履歴が変わらないことを確認する。
- ChangeSetの前後適用で、UUID、日時、表示順、および導出状態を正確に復元できることを確認する。
- Redoが元コマンドを再実行せず、元の`after`を適用することを確認する。
- 101件目、Redo分岐、および終了時に、参照中のアセットを残して未参照アセットだけを
  清掃可能にすることを確認する。
- 保存済み状態と同じ内容へ戻った場合に未保存表示が消え、再度異なる状態になると表示されることを確認する。
- 保存位置のChangeSetが履歴から消えても未保存判定が正しいことを確認する。
- 古い`expectedRevision`によるコマンド、Undo、およびRedoが状態を変更しないことを確認する。
- 履歴表示とログへ利用者入力および個人情報が含まれないことを確認する。

## 未決事項

- アセット実体の保持場所と保存確定前後の清掃順は、保存トランザクション詳細設計で確定する。
- コマンド、Undo、およびRedoのAPI DTOとエラーコードはAPI詳細設計で確定する。

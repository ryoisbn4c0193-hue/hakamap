# テスト・テストデータ・Windows検証詳細設計

## 目的

要件、ドメイン不変条件、JSON保存、Undo／Redo、ローカルAPI、React状態管理、
PixiJSの幾何処理、およびWindows配布物を、個人情報を含まない再現可能なデータで検証する。
日常的な高速検査と、Windows実機で行う上限・障害・性能検査を分離する。

## テスト階層

| 階層 | 主な対象 | 実行環境 |
| --- | --- | --- |
| Java単体 | 値オブジェクト、不変条件、幾何、採番、差分 | 各開発環境・CI |
| Java結合 | JSON Schema、Repository、保存・復旧、API | 各開発環境・CI |
| React単体 | フォーム、状態変換、Zod、通知 | 各開発環境・CI |
| React結合 | Query差分、Zustand、競合、ダイアログ | 各開発環境・CI |
| PixiJS | 座標変換、当たり判定、スナップ、リソース解放 | 各開発環境・CI |
| E2E | 主要利用者シナリオ、保存、再起動、出力 | Windows実機 |
| 障害試験 | 切断、容量不足、ロック、破損、結果不明 | Windows実機・試験共有 |
| 性能試験 | 起動、読込、検索、操作、保存、メモリ | Windows最低・推奨環境 |
| パッケージ試験 | インストール、起動、修復、アンインストール | Windows 11 |

- 単体テストで検証できる組み合わせをE2Eだけへ依存させない。
- ファイルシステムを使用する自動テストはテストごとの一時ディレクトリを使用する。
- 実際の利用者保存先、`%LOCALAPPDATA%\Hakamap`、外付け媒体、およびNASを
  通常の自動テストから読み書きしない。
- 時刻、UUID、乱数、およびファイル障害はテスト用インターフェースから制御可能にする。

## テストデータの安全性

- 氏名、戒名、墓所名、管理番号、添付表示名、および説明は架空データだけを使用する。
- 実際の墓地図、写真、戸籍・名簿由来データ、利用者Project、ログ、およびバックアップを
  テスト資産や不具合報告へ含めない。
- テスト用画像とPDFはリポジトリ内の生成スクリプトまたは安全性を確認した固定fixtureから作る。
- PDFは静的な1ページを正常系とし、複数ページ、スクリプト、フォーム、埋め込み添付などを
  含む検証用ファイルは無害な合成fixtureとして隔離する。
- 大容量・大画素fixtureは巨大なバイナリをGit管理せず、試験時に決定的に生成する。
- テスト失敗出力にも入力文字列、絶対パス、ファイル内容、および一時認証情報を出さない。

## データ規模

| 規模 | 墓所 | 人物 | 添付 | 用途 |
| --- | ---: | ---: | ---: | --- |
| 小規模 | 10 | 墓所ごとに0～3 | 墓所ごとに0～3 | 日常の機能・E2E |
| 標準規模 | 1,000 | 平均2 | 平均2 | 日常の性能劣化検知 |
| 上限規模 | 5,000 | 境界値を含む | 1墓所20件を含む | Windows重負荷・上限 |

- 上限規模では、全墓所へ20添付する極端な防御ケースと、現実的な添付分布の代表ケースを分ける。
- エリアなし、1エリア、6エリア、未割当、未採番、情報未完成、および境界上の墓所を含める。
- 負座標、小数第3位、最大文字数、正規化で同値となる文字列、辺・角接触、および
  50%の視覚的行判定を含める。
- 同じseedからUUID以外も含めて同じ論理データを再生成できるようにする。

## Java単体テスト

### ドメインと不変条件

- エリア名と業務キーの正規化、一意性、文字数、および未採番保存を検証する。
- 墓所とエリアの正の寸法、重なり禁止、辺・角接触許可、および中心点所属を検証する。
- 共有境界上でエリア表示順により所属が一意に決まることを検証する。
- 人物・添付の所有関係、表示順、上限、および墓所削除時の一括削除を検証する。
- 一括作成、範囲埋め、コピー、移動、削除、および採番が全件成功または全件失敗になることを検証する。
- 一括採番の50%行判定、行内順、行順、およびUUID同値決定順を境界値で検証する。

### コマンドと履歴

- 1コマンドが1つのChangeSetとなり、失敗とno-opで履歴、revision、日時が変化しないことを検証する。
- Undoがbefore、Redoがafterを適用し、コマンドを再実行しないことを検証する。
- 最大100件、Redo破棄、保存位置、dirtyの指紋判定、およびアセット不足時の
  全体拒否を検証する。
- 固定ClockとUUID生成器により、複数エンティティの同一コマンド日時を検証する。

## 保存・復旧結合テスト

- Project、Catalog、およびRecoveryの全fixtureを対応するJSON Schemaで検証する。
- 未知項目、`null`、不正UUID、重複参照、上限超過、破損JSON、および将来Schemaを拒否する。
- 新規アセット配置、JSON一時書込み、検証、原子的置換、不要アセット清掃の順序を検証する。
- 確定点より前の各工程へ障害を注入し、旧正式データが維持されることを確認する。
- JSON置換後の清掃失敗が保存成功となり、次回に再試行されることを確認する。
- 原子的置換結果不明時に自動再試行せず、新旧SHA-256で判定することを確認する。
- Recovery適用が正式データを上書きせず、revision 0、空履歴、未保存状態になることを確認する。
- バックアップ3世代、当日1回、復元前退避、および復元失敗時の正式データ維持を検証する。

## ローカルAPI結合テスト

- セッション、CSRF、Origin、Host、Project UUID、および`expectedRevision`を検証する。
- CommandTypeごとの専用DTO、未知項目、クライアント指定UUID、重複対象を拒否する。
- コマンド成功差分を適用した状態がスナップショット再取得結果と一致することを検証する。
- 配置確認トークン、採番プレビュートークン、および一時ファイル選択IDの
  セッション拘束、単一使用、5分期限、revision拘束を検証する。
- JSON本文、ネスト、文字列、数値トークン、および各保存JSONの防御上限を境界値で検証する。
- Problem Detailsのstatus、code、最大100違反、総件数、および個人情報非出力を検証する。
- Operationの状態遷移、約500ミリ秒ポーリング相当の取得、確定後キャンセル拒否、
  セッション拘束、および10分清掃を検証する。
- Project配下の保存・エクスポート・バックアップ復元、Catalog配下のインポート、
  ごみ箱Projectの復元がそれぞれ専用DTOだけを受け付け、未知Operationを拒否する。
- ファイル選択の単一・複数・フォルダ、用途組み合わせ、キャンセル、多重起動、
  5分期限、単一使用、および明示無効化を検証する。
- CSRFトークンが認証済み`GET /api/v1/session`のヘッダーから取得され、
  セッション再生成後に旧トークンを使用できないことを検証する。
- LANインターフェースへバインドされず、別Originおよび認証なしでAPIを使用できないことを検証する。

## React状態管理テスト

- スナップショット取得前に編集が有効にならず、取得後に確定状態が表示されることを検証する。
- API差分を1回で適用し、削除対象の選択解除と導出状態更新が行われることを検証する。
- 初回スナップショットへ人物全件が含まれず、選択墓所の人物を100件ずつ取得し、
  仮想スクロールと次ページ取得が機能することを検証する。
- 人物作成・更新・削除差分が該当墓所のキャッシュだけへ反映され、不整合時に
  先頭ページから再取得されることを検証する。
- フォーム入力中にQueryキャッシュが変化せず、成功後にJava確定値で再初期化されることを検証する。
- 未確定フォームがある選択変更で破棄・キャンセル確認が表示されることを検証する。
- revision競合時にフォーム入力を保持し、最新値の採用、再適用、コピー可能な退避を選べることを検証する。
- 地図操作、一括操作、削除、および採番が競合時に自動再実行されないことを検証する。
- Project切り替えで前Projectの選択、検索、フォーム、ダイアログ、および操作候補が消えることを検証する。
- キーボード、フォーカス、エラー関連付け、および確認ダイアログのアクセシビリティを検証する。

## PixiJSテスト

- 地図座標と画面座標の往復、負座標、10%・800%、高DPI、およびポインター中心ズームを検証する。
- 墓所の内部、辺、角、接触境界、描画順、および操作ハンドルの当たり判定を検証する。
- 四方向の矩形選択、Ctrl追加、および少しでも交差する形状の選択を検証する。
- 画面上8pxのスナップ距離がズーム率によらず維持され、角・辺・墓所・エリアの
  優先順位が安定することを検証する。
- 重なり候補と辺・角接触の表示を区別し、Java拒否後に確定形状へ戻ることを検証する。
- 背景タイルの選択、読み込み、画面外解放、およびProject切り替え時のGPUリソース解放を検証する。
- 画像比較はレイヤー順、選択表示、警告色など意味のある少数の基準画像へ限定し、
  OSフォント差に依存するラベル全体を比較しない。

## Windows E2E

主要シナリオを次の順で確認する。

1. インストールしてショートカットから起動する。
2. Projectを作成し、背景を追加する。
3. エリアと墓所を単体・一括で作成する。
4. 墓所情報、人物、および添付を登録する。
5. 検索、選択、移動、コピー、削除、Undo、およびRedoを行う。
6. 手動保存し、アプリを終了して再起動する。
7. JSONとアセットが同じ状態で復元されることを確認する。
8. PDF・画像出力、エクスポート、およびインポートを確認する。
9. バックアップ復元と自動復旧候補を確認する。
10. アンインストール時に利用者Projectを削除しないことを確認する。

- ブラウザタブを閉じてもバックエンドが終了せず、ショートカット再実行で再表示できることを確認する。
- 画面内の「Hakamapを終了」で未保存確認後にプロセスが終了することを確認する。
- 1366×768と1920×1080、Windows表示倍率100%と代表的な高DPI設定を確認する。
- インターネットを切断した状態で全シナリオを実行できることを確認する。

## NAS・外付け媒体・障害試験

- NAS試験は単一端末・単一ユーザーに限定し、設計で動作保証対象とした共有方式で行う。
- 同じProjectの二重オープン、ロック取得失敗、ロック喪失、および別プロセスによる
  外部更新を確認する。
- 読込中、アセット配置中、JSON一時書込み中、原子的置換直前・直後、および
  バックアップ中に接続を切断する。
- 再接続後に旧状態、新状態、または結果不明を安全に判定し、無条件で上書きしないことを確認する。
- 容量不足、読み取り専用、ファイル名制約、ウイルス対策ソフトによる一時占有、
  外付け媒体取り外しを確認する。
- ファイルロックまたは必要な置換方式が機能しない共有をサポート対象外として拒否・案内できることを確認する。

## 性能合否基準

| 項目 | 対象データ | 合否基準 |
| --- | --- | ---: |
| アプリ起動 | Project未指定 | 5秒以内 |
| Project読込 | 標準・上限規模 | 5秒以内 |
| 通常操作 | 選択、移動、パネル表示 | 1秒以内 |
| 検索 | 上限5,000墓所 | 1秒以内 |
| 手動保存 | 変更ありの上限規模 | 5秒以内 |
| メモリ | 上限規模の代表操作 | 1GB以内 |

- 時間は利用者が操作を開始して結果または明確な進捗表示を確認できるまでを測定する。
- 最低環境相当と推奨環境相当のWindows 11で、CPU、メモリ、保存媒体、表示解像度、
  Windowsビルド、JVM、およびアプリ版を記録する。
- ウォームアップ、測定回数、中央値、最大値を記録し、単発の最良値だけで合否判定しない。
- 最大背景、背景タイル生成、全添付検証など長時間Operationは、5秒以内完了ではなく
  300ミリ秒を超える場合の進捗表示、キャンセル、およびメモリ上限を個別に評価する。
- 1GBを超えた場合はJava heap、ブラウザ、GPU関連の内訳を採取し、単純なheap上限変更で
  合格扱いにしない。

## 実行区分

### 日常検査

- Javaの標準検査は`cd backend && ./mvnw verify`とする。
- Reactの標準検査は`cd frontend`で`pnpm lint`、`pnpm format:check`、
  `pnpm test`、`pnpm build`とする。
- 小規模fixtureを全件、標準規模fixtureを性能劣化検知に必要な範囲で使用する。
- OSや実NASを必要としないテストは日常検査で実行する。

### Windows重負荷検査

- 上限規模、最大背景、極端な添付件数、NAS、外付け媒体、切断、およびパッケージ試験を実行する。
- リリース候補ごと、および保存・描画・パッケージ方式の重要変更時に必須とする。
- 実行に時間がかかる検査を日常CIから外す場合も、手順、使用データseed、結果、
  未実施理由をリリース記録へ残す。

## 完了条件

- 要件と各詳細設計の検証要件が、単体・結合・E2E・手動試験のいずれかへ対応付けられている。
- 重大な保存・復旧・セキュリティ・不変条件の検査を手動確認だけに依存していない。
- Windows最低環境相当で性能基準を満たし、推奨環境でも退行がない。
- 未実施、失敗、環境差、および既知制約がリリース判断者へ明示されている。
- テスト資産、ログ、スクリーンショット、および報告書へ実利用者データが含まれていない。

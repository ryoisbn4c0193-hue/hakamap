# ローカルHTTP境界設計

> **状態:** 既定ブラウザとローカルHTTPサーバーを使用する現在のスキャフォールドを
> 継続する場合の候補設計。最終的なデスクトップ実行方式の選定後に確定する。

## 目的

HakamapのブラウザUIとSpring Boot間の通信を端末内に限定し、同じ端末上の
別プロセスや外部Webサイトからの意図しない操作を防止する。

## 決定事項

### ネットワーク境界

- パッケージ版のHTTPサーバーはIPv4ループバックアドレス`127.0.0.1`だけに
  バインドする。
- `0.0.0.0`および端末のLANアドレスにはバインドしない。
- 名前解決結果に依存しないよう、パッケージ版の接続先には`localhost`ではなく
  `127.0.0.1`を使用する。
- パッケージ版では固定ポートを使用せず、起動時にOSが利用可能なポートを選択する。
- 開発時は`127.0.0.1:8080`をSpring Bootの既定の接続先として使用してよい。
- Viteの開発サーバーもループバックアドレスだけにバインドする。

### UIとAPIの配信

- パッケージ版では、Reactの本番ビルドと`/api`を同じSpring Bootプロセスから配信する。
- UIとAPIを同一オリジンにし、パッケージ版でCORSを有効にしない。
- 開発時に限り、ViteからSpring Bootの`/api`へプロキシしてよい。
- 実行時に外部のWebサーバー、CDN、地図タイル、認証サービスへ依存しない。

### ローカルアクセスの認証

- `127.0.0.1`からのアクセスであることだけを認証として扱わない。
- バックエンドの起動ごとに、推測が困難な一時的な認証情報を生成する。
- ランチャーが開いたブラウザだけが、その認証情報を使用してアプリケーションの
  セッションを開始できるようにする。
- セッション開始後の認証状態は、JavaScriptから読み取れない`HttpOnly` Cookieで管理する。
- Cookieには少なくとも`SameSite=Strict`を設定する。
- 起動用の認証情報を受け取った後は、それをブラウザのアドレス欄や通常の画面URLに
  残さないよう、認証情報を含まないURLへリダイレクトする。
- 起動用の認証情報をアプリケーションログへ記録しない。
- 状態を変更するAPIには、セッション認証に加えてCSRF対策を適用する。
- OriginやHostなど、ブラウザが送信するリクエスト情報も補助的に検証する。

### 二重起動との連携

- 起動中のポート番号と、既存プロセスへ接続するために必要な情報を、現在の
  Windowsユーザーだけがアクセスできるアプリケーションデータ領域へ保存する。
- 起動情報に記録されたプロセスが実在し、Hakamapとして応答することを確認してから
  既存プロセスとして再利用する。
- 古い起動情報だけが残っている場合は、安全に無効化して新しいプロセスを起動する。
- 正常終了時は起動情報を削除する。
- 起動用認証情報やセッションはバックエンドプロセスの終了時に無効になるものとし、
  永続的なユーザー認証情報として保存しない。

## 開発環境

- 開発時もSpring BootとViteをループバックアドレスだけにバインドする。
- Viteから`/api`へプロキシし、アプリケーションコードに開発用API URLを埋め込まない。
- 開発用の認証省略が必要になった場合は、明示的な開発プロファイルだけに限定し、
  パッケージ版で有効にならないことをテストする。

## 検証要件

- サーバーソケットが`127.0.0.1`だけで待ち受けていることを統合テストまたは
  パッケージ試験で確認する。
- LAN上の別端末から接続できないことをWindowsパッケージ試験で確認する。
- 認証情報がないリクエストでは、非公開データの取得と状態変更ができないことを確認する。
- 外部オリジンを想定した状態変更リクエストが拒否されることを確認する。
- 2回目のランチャー起動で新しいバックエンドが作られず、既存の画面が開くことを確認する。
- 古い起動情報が残った状態から正常に復旧できることを確認する。

## 採用しない方式

- `0.0.0.0`へのバインド
- 制限のないCORS
- ループバック接続であることだけを根拠とする無認証API
- 固定された認証トークンの永続利用
- 認証情報をURLへ残したまま通常画面を利用する方式

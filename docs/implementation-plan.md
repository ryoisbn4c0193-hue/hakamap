# Hakamap MVP実装計画

## 目的

レビュー合格済みの要件・基本設計・詳細設計を、データ安全性を維持しながら
検証可能な小さな単位で実装する。ドメインと保存基盤を先行し、その後にローカルHTTP、
React、PixiJS、ファイル処理、Windows配布を段階的に統合する。

## 現在地

- 要件定義、基本設計、詳細設計8工程、および詳細設計レビュー指摘の反映は完了している。
- Phase 0からPhase 3を完了し、ビルド・品質基盤、Projectドメイン、JSON保存モデル、
  原子的保存、編集履歴、およびRecoveryスナップショットを実装した。
- Phase 4以降のAPI、製品機能、ローカルHTTP保護、および
  Windowsパッケージは未実装である。

## 実装原則

- Javaのドメイン状態と業務判定を正とし、ReactやPixiJSへ同じ業務ルールを複製しない。
- 先に不変条件と保存形式を実装し、その上にAPIとUIを接続する。
- 各フェーズは単体テスト、結合テスト、標準検査、および文書更新まで完了してから次へ進む。
- 保存、復旧、インポートなどの破壊的処理は、正常系だけでなく確定点前後の失敗を同じフェーズで検証する。
- UIだけの仮データ実装を長期間残さず、縦切り機能ではJava APIまで接続して完了させる。
- 新しい依存関係は採用理由、Java 21・Windows・オフライン互換性、およびライセンスを確認する。
- 生成物、ユーザーデータ、テスト用大容量バイナリをGitへ含めない。

## コミットとレビュー運用

- 実装、標準検査、関連文書、および引継ぎメモをPhase単位で1つの完了コミットにまとめる。
- Phase内の実装単位ごとに作業内容と検査結果を報告するが、原則として途中コミットを作らない。
- Phaseの完了条件を満たさない状態では完了コミットを作らず、未完了事項を引継ぎメモへ記録する。
- Phaseコミットはローカルへ作成し、通常はその時点でプッシュしない。
- 外部ChatGPTレビューの節目で、レビュー対象のPhaseコミットまでを`origin/main`へプッシュする。
- プッシュ後はローカル`HEAD`と`origin/main`を照合し、レビュー対象のコミットハッシュを
  利用者へ明示する。
- レビュー指摘の修正は対象Phaseの補正コミットとして作成し、再レビュー時は補正後の
  コミットハッシュを新しいレビュー対象として伝える。

## Phase 0: ビルド・依存・品質基盤

### 実装

- H2、Spring Data JPA、およびFlywayの本番・テスト依存を`pom.xml`から除去する。
- Draft 2020-12対応のJSON Schema検証ライブラリと必要最小限のJackson依存を追加する。
- JSON Schema 3ファイルをclasspathから読み込むスモークテストを追加する。
- バックエンドのパッケージ境界を`domain`、`application`、`infrastructure`、`api`へ準備する。
- フロントエンドへMUI、Emotion、Zustand、TanStack Query、Zod、PixiJSを追加する。
- 製品テーマ、QueryClient、エラー境界、テスト用Providerの最小構成を作る。
- Viteの`/api`プロキシを`127.0.0.1`の開発用バックエンドへ限定する。
- Mavenから固定Node.js・pnpmでフロントエンドを検査・ビルドし、JARへ同梱する方式を実装する。
- `target`、`dist`、`node_modules`などの生成物がGit管理されないことを再確認する。

### 完了条件

- `backend && ./mvnw verify`が成功する。
- `frontend`で`pnpm lint`、`pnpm format:check`、`pnpm test`、`pnpm build`が成功する。
- Maven本番ビルドだけでReactを含む実行可能JARを生成できる。
- JARからReact画面とAPIの404を同一オリジンで確認できる。
- H2、JPA、Flywayの依存と自動設定が実行時classpathに存在しない。

## Phase 1: ドメインモデルと不変条件

### 実装

- UUID、日時、名称、管理番号、地図座標、長方形、背景配置の値オブジェクトを実装する。
- Project、Area、Grave、Person、AssetMetadataのエンティティとProject集約を実装する。
- 文字列正規化、エリア名・業務キー一意性、完成状態、警告を実装する。
- 墓所・エリアの重なり、辺・角接触、墓所中心点による所属判定を実装する。
- 単体・行列・範囲埋め生成、複数移動・コピー・削除、一括採番順を実装する。
- ドメイン例外を安定した内部エラーコードへ対応付ける。

### 完了条件

- `domain-model.md`の全不変条件が単体テストへ対応付く。
- 境界値、50%行判定、Unicode正規化、原子的な一括失敗をテストできる。
- Spring、JSON、ファイルシステムへ依存せずドメインテストを実行できる。

## Phase 2: JSON保存モデルとRepository

### 実装

- Project、Catalog、Recovery v1のJackson保存recordとMapperを実装する。
- JSONストリーム防御値とSchema検証サービスを実装する。
- Schema後のJava整合性検証一覧を検証サービスとして実装する。
- 決定的なプロパティ・Map順、任意値省略、UTC日時、小数第3位を実装する。
- Project、Catalog、RecoveryのRepository interfaceとファイル実装を作る。
- 一時ディレクトリを使用するfixture・不正JSON・参照不整合テストを追加する。

### 完了条件

- v1 fixtureが読込・再保存・再読込で同じドメイン状態になる。
- Schema違反とJava整合性違反を区別して全体拒否できる。
- 絶対パスをProject・Recovery・API用モデルへ混入させない。

## Phase 3: 保存トランザクションと履歴

### 実装

- Projectロック、外部変更検知、ステージングアセット、原子的JSON置換を実装する。
- 保存確定点、結果不明、清掃再試行、空き容量不足を実装する。
- ChangeSet、最大100件のUndo／Redo、保存指紋、dirty判定を実装する。
- Recoveryスナップショットの定期作成・候補検証・未保存状態での適用を実装する。
- Clock、UUID、ファイル操作、障害注入を差し替え可能にする。

### 完了条件

- 各確定工程への障害注入で旧正式データが失われない。
- 保存後もUndoでき、保存済み状態へ戻るとdirtyが解除される。
- 結果不明時に自動再試行せず、安全に編集停止できる。

## Phase 4: アプリライフサイクルとHTTP保護

### 実装

- `127.0.0.1`限定・ランダムポートでSpring Bootを起動する。
- application lock、instance情報、制御トークン、二重起動時の再表示を実装する。
- 起動用認証、HttpOnly Cookie、`GET /api/v1/session`、CSRFヘッダーを実装する。
- Host、Origin、`Sec-Fetch-Site`、Referer、`Cache-Control: no-store`を検証する。
- Problem Details、ログの秘匿化、終了処理を実装する。

### 完了条件

- 外部Origin、LAN、未認証、旧CSRF、二重起動偽装を結合テストで拒否できる。
- Originなしの正常な同一オリジンGETでCSRFを取得できる。
- セッション再生成後に古い応答と変更要求を再利用できない。

## Phase 5: Project管理とファイル選択

### 実装

- Catalog一覧、Project作成・open・close・デフォルト・再関連付けを実装する。
- ごみ箱移動、元場所または指定場所への復元、完全削除を実装する。
- Swing `JFileChooser`をEDTで起動するファイル選択サービスを実装する。
- 選択モード、用途、5分期限、単一使用、キャンセル、多重起動拒否を実装する。
- 初回のProject管理画面と未保存切り替え確認をReactで接続する。

### 完了条件

- Projectを作成、一覧、開く、閉じる、再関連付け、ごみ箱復元できる。
- APIとログへ絶対パスを返さない。
- ファイル選択IDを別用途、別セッション、期限後に使用できない。

- Undo、Redo、履歴、人物ページング、アセット配信APIを実装する。

### 完了条件

- 成功差分を適用した状態がスナップショット再取得と一致する。

### 完了条件

- 添付追加、並べ替え、削除、画像・1ページPDF由来画像のプレビューを実装する。

## Phase 10: バックアップ・入出力

### 実装

- 手動保存後の1日1回・3世代自動バックアップを実装する。
- バックアップ一覧、不透明ID、利用可否、`backupVersion`を実装する。
- 復元前退避とバックアップ復元Operationを実装する。
- `.hakamap`のストリーミングエクスポート・インポートと防御上限を実装する。

### 完了条件

## Phase 11: Windows配布・性能・障害検証

### 実装・検証

- `jlink`と`jpackage`でJava 21ランタイム同梱のユーザー単位EXEを作成する。
- ショートカット、二重起動、再表示、画面内終了、アップグレード、アンインストールを確認する。
- Windows最低・推奨環境で起動、読込、検索、操作、保存、メモリを測定する。
- 最大背景、上限Project、外付け媒体、NAS、切断、容量不足、ロック喪失を検証する。
- オフライン状態、1366×768、高DPI、ウイルス対策ソフトによる占有を確認する。

### 完了条件

- `test-strategy.md`のWindows重負荷検査と性能基準を満たす。
- サポート外の共有方式を安全に拒否または案内できる。
- 既知制約と未実施試験がない、またはリリース判断へ明記されている。

## 最初の実装単位

最初はPhase 0を次の3つの独立した変更に分ける。

1. バックエンドからH2・JPA・Flywayを除去し、JSON Schema検証のスモークテストを追加する。
2. フロントエンドへ標準依存、テーマ、Provider、および最小画面骨格を追加する。
3. Mavenとpnpmの統合ビルド、JAR同梱、同一オリジン配信を追加する。

各変更は単独で標準検査を通せる状態を維持し、Phase 0の3変更と文書更新をPhase完了時に
1コミットへまとめる。Phase 1以降はドメイン機能単位の縦切りで作業し、同じく
Phase完了時にコミットする。
